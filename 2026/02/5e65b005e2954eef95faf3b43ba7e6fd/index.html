重试和无限循环

graph TB
    subgraph "重试控制流程"
        A[节点开始] --> B{硬限制检查
_check_and_increment_node_loop}
        B -->|超限| C[返回 FAILED/NEEDS_HUMAN]
        B -->|通过| D[执行节点逻辑]
        
        D --> E{执行成功?}
        E -->|异常| F[进入 Error Handler]
        E -->|成功| G[调用决策函数]
        
        G --> H{决策结果}
        H -->|SUCCESS/VALID| I[重置软限制计数]
        I --> J[继续下一节点]
        
        H -->|NEEDS_RETRY| K{软限制检查
_can_retry_stage}
        K -->|未超限| L[重试计数+1]
        L --> A
        
        K -->|已超限| M[降级为 NEEDS_REPAIR]
        M --> N[应用修复参数]
        N --> A
    end
    
    subgraph "计数器说明"
        O["硬限制计数器 (node_current_loops)
• 每次进入 +1
• 永不重置
• 防止死循环"]
        P["软限制计数器 (stage_current_retries)
• 仅 NEEDS_RETRY 时 +1
• 成功时重置
• 控制质量重试"]
        Q["全局计数器 (global_current_loops)
• 每次节点进入 +1
• 所有节点共享
• 最后防线"]
    end
    
    style B fill:#FF5722,color:white
    style K fill:#FF9800,color:black
    style H fill:#FFEB3B,color:black
机制 计数器 触发条件 重置条件 作用
硬限制 node_current_loops 每次进入节点 永不重置 防止死循环
软限制 stage_current_retries 决策返回 NEEDS_RETRY 成功时重置为0 控制质量重试
全局后备 global_current_loops 每次进入节点 永不重置 最后防线

关键设计原则：

  1. 软限制失败后降级为 NEEDS_REPAIR，不是直接失败
  2. 修复参数会随 NEEDS_REPAIR 传递，指导节点如何修复
  3. 全局限制作为兜底，防止极端情况下的无限循环

双重保护机制

graph TB
    subgraph "进入节点"
        A[节点执行] --> B[_check_and_increment_node_loop
硬限制检查]
    end
    
    subgraph "硬限制 (Node Loop Control)"
        B --> C{节点循环次数
≥ node_max_loops?}
        C -->|是| D[FAILED / NEEDS_HUMAN]
        C -->|否| E[继续执行节点逻辑]
    end
    
    subgraph "节点执行"
        E --> F[节点业务逻辑]
        F --> G{执行成功?}
        G -->|异常| H[进入错误处理]
        G -->|成功| I[决策函数]
    end
    
    subgraph "软限制 (Stage Retry Control)"
        I --> J{质量问题?}
        J -->|需要重试| K{stage_current_retries
>= stage_max_retries?}
        K -->|否| L[NEEDS_RETRY + 重试计数+1
重新执行节点]
        K -->|是| M[NEEDS_REPAIR / NEEDS_HUMAN]
        J -->|成功| N[重置重试计数=0
继续下一节点]
    end
    
    subgraph "全局后备"
        O[全局循环检查] --> P{global_current_loops
>= global_max_loops?}
        P -->|是| Q[强制失败]
    end
    
    style C fill:#FF5722,color:white
    style K fill:#FF9800,color:black
    style P fill:#F44336,color:white

计数变化时序图

sequenceDiagram
    participant Node as 节点
    participant Hard as 硬限制计数器
    participant Soft as 软限制计数器
    
    Note over Node,Soft: 第一次执行
    Node->>Hard: +1 (0→1)
    Node->>Soft: 0 (初始)
    Node->>Node: 执行失败，需要重试
    Node->>Soft: +1 (0→1)
    Node-->>Node: 返回 NEEDS_RETRY
    
    Note over Node,Soft: 重试执行
    Node->>Hard: +1 (1→2)
    Node->>Soft: 1 (不变)
    Node->>Node: 执行成功
    Node->>Soft: 重置 (1→0)
    Node-->>Node: 继续下一节点
    
    Note over Node,Soft: 后续因修复再次进入
    Node->>Hard: +1 (2→3)
    Node->>Soft: 0→1 (新周期)
    Node->>Node: 成功
    Node->>Soft: 重置 (1→0)
    
    Note over Hard: 硬限制持续累积

决策状态流转图

stateDiagram-v2
    [*] --> 节点执行
    
    节点执行 --> 成功: 业务逻辑成功
    节点执行 --> 异常: 抛出异常
    
    成功 --> 决策函数
    异常 --> 错误处理
    
    决策函数 --> SUCCESS: 无质量问题
    决策函数 --> VALID: 轻微问题，可接受
    决策函数 --> NEEDS_RETRY: 有问题，可重试
    决策函数 --> NEEDS_REPAIR: 需修复后重试
    决策函数 --> NEEDS_HUMAN: 需人工干预
    决策函数 --> FAILED: 严重失败
    
    NEEDS_RETRY --> 检查软限制
    
    检查软限制 --> 节点执行: 未超限 (current < max)
    检查软限制 --> NEEDS_REPAIR: 已超限
    
    NEEDS_REPAIR --> 节点执行: 应用修复参数后重试
    
    SUCCESS --> [*]: 重置软限制，继续流程
    VALID --> [*]: 重置软限制，继续流程
    NEEDS_HUMAN --> [*]: 等待人工决策
    FAILED --> [*]: 进入错误处理
    
    错误处理 --> 节点执行: 可恢复错误，重新开始
    错误处理 --> NEEDS_HUMAN: 不可恢复
    错误处理 --> [*]: 中止流程
