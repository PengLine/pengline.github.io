基于OpenSearch + Milvus的混合召回策略

发表于2026-07-08|更新于2026-07-12
|架构推荐系统
|浏览量:

OpenSearch + Milvus

  • 推荐系统架构总览（系统总体架构、多路召回、混合检索、多模态融合、重排策略）
  • 阶段一：基于文本向量（文本向量化、BERT推理、双路召回、规则排序、存储架构）
  • 阶段二：基于多模态融合（三模态特征提取、动态权重融合、粗排层、GBDT精排）
  • 阶段三：基于个性化精排（实时行为流、滑动窗口向量、DIN模型、自适应E&E）
  • 各阶段算法策略（召回、粗排、精排、重排，协同过滤、向量双塔、热门召回、MMoE与PLE、线性加权、滑动窗口、MMR）
  • 多路并行召回策略（I2I召回、U2I召回、热搜召回、探索召回、向量检索、协同过滤）
  • 用户画像构建（显式与隐式画像融合、标签画像与向量画像双轨构建、滑动窗口、行为序列建模）

为什么需要 Milvus？

OpenSearch 的能力边界

在前三阶段的架构中，OpenSearch承担了向量检索和关键词检索的双重职责。对于千万级视频库、QPS在数千级别的场景，这套架构运行良好。但当我们把视野放到亿级视频库、万级QPS的工业级规模时，OpenSearch的向量检索能力开始显现出瓶颈。

OpenSearch的核心设计目标是全文检索与分析，向量检索是通过k-NN插件附加的能力：

  • 索引构建效率：HNSW索引构建在OpenSearch中需要全量数据扫描，亿级数据的索引构建可能需要数小时甚至数天
  • 内存占用：OpenSearch的k-NN索引将全部向量加载到堆外内存，亿级768维向量需要约200GB内存，且难以水平扩展
  • 混合查询性能：虽然OpenSearch 3.0引入了common filter支持，但在高过滤率场景下，向量检索性能会显著下降

Milvus 的核心优势

Milvus是专为AI原生场景设计的开源向量数据库，在十亿级向量规模下提供高并发、低延迟的向量检索能力。与OpenSearch相比，Milvus在以下维度具有显著优势：

能力维度 OpenSearch Milvus 差距说明
向量规模 千万级 十亿级 Milvus专为大规模设计
索引类型 HNSW为主 HNSW/IVF/IVF_PQ/DiskANN Milvus支持更多索引算法
标量过滤 强（倒排索引） 中（标量索引） OpenSearch的倒排索引在复杂过滤上更强
混合查询 原生支持 需应用层协调 OpenSearch的优势领域
GPU加速 有限支持 原生支持 Milvus支持GPU IVF索引
内存优化 有限 PQ量化压缩 Milvus可压缩至原始大小的1/8-1/32

混合架构的定位

基于以上分析，我们得出一个关键结论：OpenSearch和Milvus并非替代关系，而是互补关系。具体来说：

  • OpenSearch：负责处理需要复杂标量过滤（多字段组合、范围查询）和关键词匹配的召回场景。当过滤条件复杂、候选集较小时，OpenSearch的倒排索引效率更高。

  • Milvus：负责处理大规模纯向量检索或简单过滤的召回场景。当向量数据量超过千万级、过滤条件简单时，Milvus的性能优势明显。

对比维度 OpenSearch Milvus
核心定位 搜索引擎 + 向量附加能力 专用向量数据库
向量索引算法 HNSW、IVF（Faiss 引擎） HNSW、IVF_FLAT、IVF_SQ8、IVF_PQ、DiskANN、CAGRA 等
量化压缩 sq、pq（Faiss 引擎） sq、pq、sq8、rabiq 等
GPU 加速 有限支持（Faiss 引擎） 原生支持（GPU_IVF_FLAT、GPU_CAGRA 等）
混合检索 原生支持（倒排索引 + k-NN） 通过标量索引实现，但全文检索需外部集成
增量索引 需触发分片合并 支持实时增量构建
多向量字段 单文档仅支持 1 个向量字段 支持 1 个 Collection 多向量字段
内存管理 向量与全文索引共享内存 独立管理，支持冷热分离

OpenSearch 向量检索原理

OpenSearch 本质上是搜索引擎，向量检索是通过 k-NN 插件附加的能力。其核心设计目标依然是全文检索与数据分析，向量检索是“锦上添花”的功能，而非根本设计。这种定位决定了 OpenSearch 的向量检索必然与其倒排索引体系深度耦合，在混合检索场景下有独特优势，但在纯向量检索场景下受限于其底层架构。

k-NN 向量字段与索引构建

OpenSearch 通过 knn_vector 字段类型来存储向量数据。要启用向量检索功能，需要在索引设置中将 index.knn 参数设为 true。每个文档的向量字段与文档的其他字段一起存储，共享同一个 Lucene 分片。

向量索引算法与引擎

OpenSearch 的 k-NN 插件支持三种底层引擎，每种引擎适用不同场景：

引擎 特点 适用场景
Lucene OpenSearch 原生引擎，与全文检索深度集成 混合检索场景，数据量中等
nmslib 高性能向量库，HNSW 算法实现成熟 追求低延迟的向量检索，数据量较小
Faiss Meta 开源的向量检索库，功能最丰富 支持量化压缩、GPU 加速，适合大规模场景

Faiss 引擎的核心优势在于支持多种量化编码器，可以显著降低内存占用：

  • flat（无压缩）：保持原始精度，内存占用最大

  • sq（标量量化，如 sq8）：将每个向量维度从 4 字节压缩为 1 字节，内存减少 75%，精度损失小

  • pq（乘积量化）：将向量分割为子向量分别编码，可实现 4-32 倍压缩，适合内存受限场景

OpenSearch 从 2.9 版本开始支持 mode 参数来配置向量检索模式：

in_memory 模式（默认）：使用 nmslib 引擎，无量化压缩，优先低延迟检索。适合对延迟敏感、数据量可控的场景。

on_disk 模式：使用 Faiss 引擎，默认启用 32x 压缩，采用“两阶段检索”——先用量化索引快速筛选候选集，再用原始精度重排序。适合大规模向量存储，在内存成本和召回率之间取得平衡。

精确 k-NN 与脚本评分

除了近似 k-NN（ANN）检索，OpenSearch 还支持精确 k-NN 检索，通过 script_score 查询实现。这种方式对全量数据进行暴力遍历计算距离，保证 100% 召回率。

关键优势是先过滤再精确计算，适合过滤后候选集较小的场景。但当过滤后数据量超过一万条时，延迟会急剧上升，因此主要用于小规模精确匹配场景。

SIMD 硬件加速

OpenSearch 从 2.13 版本开始支持 SIMD（单指令多数据流）优化，利用 CPU 的 AVX2、AVX512（x64 架构）或 Neon（ARM64 架构）指令集并行计算向量距离，显著提升检索吞吐量。该优化仅在向量维度为 8 的倍数时生效，且仅支持 Linux 系统上的 Faiss 引擎。在生产环境中部署时，建议优先选择支持 AVX512 指令集的 CPU 实例，以获得最优性能。

Milvus 向量检索原理

Milvus 是专为向量检索设计的原生向量数据库。其核心设计目标就是在十亿级向量规模下提供高并发、低延迟的检索能力。这决定了 Milvus 从底层存储引擎到查询执行器，全部围绕向量的高效存储和相似性计算进行优化。

向量索引的三层架构

Milvus 的向量索引由三个核心部件组成：数据结构、量化和精炼器。这种分层设计使其能够在精度和性能之间灵活权衡。

数据结构是索引的基础层，决定向量如何被组织和索引：

  • **IVF（倒排文件）**通过 K-Means 聚类将向量划分到不同的桶（bucket）中。检索时只扫描中心点距离查询向量最近的若干个桶，而非全量数据，大幅降低计算成本。适合需要高吞吐量的大规模数据集。其时间复杂度从 O(n) 降至 O(V + n/V × k)，其中 V 为聚类数，k 为扫描的桶数。

  • **HNSW（分层导航小世界）**构建多层图结构：上层节点稀疏，提供粗粒度导航；下层节点密集，保证检索精度。查询从顶层开始，逐层向下搜索，实现对数时间的检索复杂度。适合高维空间和低延迟要求的场景。

量化通过更粗略的向量表示来减少内存占用和计算成本：

  • **标量量化（SQ）**将每个浮点维度压缩为 8 位整数，内存减少 75%，精度损失小。例如，sq8 将 32 位浮点数映射到 256 个整数级别，牺牲少量精度换取显著的内存收益。

  • **乘积量化（PQ）**将高维向量分割成 M 个子向量，每个子向量在独立的码本中编码。可实现 4-32 倍压缩，适合内存极度受限的超大规模场景。某实验显示 128 维向量经 PQ 压缩后存储效率提升 16 倍。

  • 精炼器用于补偿量化带来的精度损失。量化检索时，系统会返回 topK × expansion rate 个候选向量，然后精炼器用更高精度（如 FP32）重新计算这些候选向量的距离，从中选出最终的精确 Top-K。这种“粗排+精排”的两阶段策略在速度和精度之间取得了最佳平衡。

场景 推荐索引 说明
原始数据全部在内存中 HNSW / IVF + 精炼 HNSW 适合低延迟、高召回率场景
数据量超大，内存有限 IVF_PQ + mmap PQ 压缩 + 内存映射，平衡内存与磁盘访问
数据在 SSD 上，对延迟敏感 DiskANN 索引部分驻留磁盘，稳定延迟
高过滤率（>95%） FLAT（暴力检索） 候选集很小，索引开销反而增加
极高召回率（>99%） FLAT + GPU GPU 暴力遍历，无精度损失

标量字段索引：除了向量索引，Milvus 2.1.0 之后支持为标量字段建立索引，加速“向量检索 + 标量过滤”的混合查询-1。不同数据类型有对应的索引算法：

数据类型 推荐索引 说明
INT / VARCHAR 倒排索引 点查询加速 30 倍
BOOL BITMAP 位图索引
ARRAY BITMAP 数组元素快速匹配

标量索引与过滤

Milvus从2.1.0版本开始支持标量字段索引，通过倒排索引或BTREE索引加速标量过滤。这一能力使得Milvus在“向量检索+标量过滤”的组合场景中更加高效。

标量索引算法：

数据类型 推荐索引 说明
INT8/16/32/64 倒排索引 点查询加速30倍
VARCHAR 倒排索引 支持模式匹配
FLOAT/DOUBLE 倒排索引 范围查询加速
BOOL BITMAP 位图索引

性能数据：在100万条记录的数据集上，使用倒排索引可为点查询提供高达30倍的性能提升。

检索参数调优

关键参数对检索性能影响显著：

  • nprobe（IVF 检索聚类数）：指定访问多少个聚类桶。增大可提高召回率，但增加计算量。初始值建议设为 nlist / 10。

  • ef_search（HNSW 搜索范围）：控制 HNSW 图搜索的候选节点数。增大可提高召回率，但增加延迟。

  • search_k（PQ 重排序候选数）：指定精炼器处理的候选数量。增大可提高最终召回率，但增加精炼开销。

双引擎协同的召回策略

策略一：OpenSearch主导的复杂过滤召回

适用场景：用户筛选条件复杂（如“只看音乐类、近7天发布、时长3-5分钟、4K画质的视频”），需要多字段组合过滤。

优势：OpenSearch的倒排索引对多字段组合查询天然支持，效率高于Milvus的标量索引。

flowchart LR
    Q[复杂过滤条件] --> OS[(OpenSearch)]
    OS --> F[倒排索引过滤]
    F --> V[k-NN向量检索]
    V --> R[Top 100召回结果]

策略二：Milvus主导的大规模向量召回

适用场景：纯向量相似度检索或仅有简单过滤（如“只看未观看过的视频”），数据量超过5000万。

优势：Milvus的向量检索性能在亿级数据上远超OpenSearch，且支持GPU加速。

flowchart LR
    Q[用户向量] --> ML[(Milvus)]
    ML --> I[向量索引检索]
    I --> F[标量过滤]
    F --> R[Top 100召回结果]

策略三：双引擎融合召回

适用场景：需要同时兼顾复杂过滤和大规模向量检索，对召回质量要求最高。

flowchart TB
    Q[用户请求] --> Split[请求拆分]
    
    Split --> |路线1| OS[(OpenSearch)]
    Split --> |路线2| ML[(Milvus)]
    
    OS --> |复杂过滤+向量| R1[召回结果1]
    ML --> |纯向量/简单过滤| R2[召回结果2]
    
    R1 --> Merge[结果融合]
    R2 --> Merge
    
    Merge --> Dedup[去重]
    Dedup --> Norm[分数归一化]
    Norm --> Rank[加权排序]
    Rank --> Final[最终结果]

融合策略：

  1. OpenSearch召回100条（处理复杂过滤）

  2. Milvus召回100条（处理大规模向量检索）

  3. 合并去重，得到约150-180条

  4. 分数归一化：使用sigmoid变换统一分数尺度

  5. 加权融合： final_score = 0.6 × OpenSearch_score + 0.4 × Milvus_score 或根据场景动态调整权重

总结

引入Milvus后，推荐系统架构从“单引擎”演进为“双引擎协同”，核心设计原则：

  1. 引擎专长化：OpenSearch负责复杂标量过滤和混合检索，Milvus负责大规模向量检索和GPU加速。两者各司其职，发挥各自优势。

  2. 查询路由智能化：根据过滤条件复杂度、数据规模、QPS要求等维度，动态决策使用哪个引擎，实现最优的延迟和召回效果。

  3. 数据同步一致性：通过双写+定时对账保证两引擎数据一致性，同时设计降级策略保证高可用。

  4. 渐进式迁移：从探索验证到双轨并行再到切流完成，分阶段推进，控制迁移风险。

  5. 索引选型适配：根据数据规模、内存约束、召回率要求选择最合适的Milvus索引类型，在精度和性能之间取得平衡。
