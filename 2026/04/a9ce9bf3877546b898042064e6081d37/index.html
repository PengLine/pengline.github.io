分布式任务调度（NeoTask）持久化

NeoTask 持久化

📚 返回首页 | 📖 快速上手 | 📘 API 参考 | ⚙️ 配置说明 | 🐙 GitHub | 🐱 NeoTask 官网

概述

在分布式任务调度系统中，持久化能力是保障任务不丢失的基石。NeoTask 设计了多层次的持久化架构，支持从单机开发到大规模分布式部署的平滑演进。

任务持久化主要解决三个核心问题。其一是任务不丢失，即使系统崩溃或节点故障，已提交的任务也能够被恢复和执行。其二是状态可追溯，每个任务的生命周期都有完整的状态记录，便于问题排查和审计。其三是存储可扩展，根据业务规模和性能要求，可以选择不同的存储后端。

NeoTask 的持久化设计遵循存储抽象原则。通过定义统一的存储接口，将任务数据的操作与具体存储实现解耦。这种设计使得系统可以在内存、SQLite、Redis 等多种存储后端之间灵活切换，而核心业务逻辑无需任何修改。

持久化架构

持久化架构由存储抽象层、存储实现层和存储工厂层三个层次组成。

存储抽象层定义了任务存储和队列存储的标准接口。任务存储接口负责任务的创建、查询、更新、删除和列表操作，队列存储接口负责队列的入队、出队、大小查询和清空操作。这些接口是所有存储实现必须遵守的契约。

存储实现层提供了多种存储后端的具体实现。内存存储适用于单元测试和开发环境，提供最快的读写速度但不持久化。SQLite 存储适用于单机生产环境，数据持久化到本地文件，无需额外部署服务。Redis 存储适用于分布式生产环境，支持多节点共享数据和水平扩展。

存储工厂层根据配置参数自动创建相应的存储实例。工厂模式封装了存储对象的创建逻辑，调用方无需关心具体使用哪种存储后端，只需要提供配置即可。

任务存储设计

任务存储负责管理任务元数据的持久化。每个任务在存储中保存为一条独立的记录，包含任务标识、任务数据、执行状态、优先级、归属节点、重试次数、生存时间、创建时间、开始时间、完成时间、执行结果和错误信息等字段。

任务标识是任务的主键，由时间戳和随机数组合生成，保证全局唯一性。任务数据存储用户提交的业务数据，以 JSON 格式序列化后保存。执行状态记录任务当前所处的阶段，是状态机转换的依据。

归属节点记录当前执行或最后一次执行该任务的节点标识，用于故障恢复时的归属判断。重试次数记录任务的失败重试次数，用于判断是否超过最大重试限制。生存时间定义了任务在终端状态下的保留时长，超期任务会被自动清理。

创建时间、开始时间和完成时间分别标记任务生命周期的关键节点，用于性能分析和延迟统计。执行结果存储任务成功执行后的返回值，错误信息存储任务失败时的异常详情。

队列存储设计

队列存储负责管理待执行任务的排队顺序。与任务存储不同，队列存储更关注性能和原子性操作。

队列的核心数据结构是有序集合。每个队列元素包含任务标识和优先级分数，分数越小优先级越高。系统根据优先级分数对任务进行排序，确保高优先级任务优先被消费。

队列操作需要保证原子性。入队操作将任务标识和优先级分数添加到有序集合中。出队操作需要同时完成三个动作：获取最高优先级的任务、从集合中移除该任务、返回任务标识。这三个动作必须在同一个原子操作中完成，否则可能出现任务被重复获取的问题。

批量出队操作支持一次获取多个任务，减少网络往返次数。批量出队的原子性同样重要，所有被取出的任务必须同时从集合中移除，不能被其他节点重复获取。

内存存储实现

内存存储是最简单的存储实现，适用于开发测试和单机低负载场景。

内存存储将所有数据保存在进程内存中，读写速度最快，没有网络开销和磁盘 I/O。但缺点也同样明显，进程重启后所有数据都会丢失，不适合生产环境使用。

内存任务存储使用字典作为底层数据结构，任务标识作为键，任务对象作为值。为了保证并发安全，所有读写操作都使用异步锁进行保护。

内存队列存储使用堆作为底层数据结构。堆中的每个元素是优先级、序列号和任务标识的三元组。序列号用于保证相同优先级的任务按照先入先出的顺序处理，避免饥饿问题。

sequenceDiagram
    participant Client as 调用方
    participant Repo as MemoryTaskRepo
    participant Lock as 异步锁
    participant Dict as 字典
    
    Client->>Repo: save(task)
    Repo->>Lock: acquire()
    Lock-->>Repo: 获取锁
    Repo->>Dict: tasks[task_id] = task
    Repo->>Lock: release()
    Lock-->>Repo: 释放锁
    Repo-->>Client: 保存成功
    
    Client->>Repo: get(task_id)
    Repo->>Lock: acquire()
    Lock-->>Repo: 获取锁
    Repo->>Dict: task = tasks.get(task_id)
    Repo->>Lock: release()
    Lock-->>Repo: 释放锁
    Repo-->>Client: 返回任务

SQLite 存储实现

SQLite 存储是单机生产环境的推荐选择。SQLite 是嵌入式关系型数据库，无需单独部署服务进程，数据持久化到本地文件。

SQLite 任务存储使用 tasks 表存储任务数据。表结构包含任务标识、任务数据、执行状态、优先级、归属节点、重试次数、生存时间、创建时间、开始时间、完成时间、执行结果和错误信息等字段。为了提高查询效率，在状态字段和优先级字段上创建了索引。

SQLite 队列存储使用 queue 表存储待执行任务。表结构包含任务标识、优先级和创建时间三个字段。出队操作通过查询优先级最小、创建时间最早的记录实现，返回后立即删除该记录。

SQLite 存储的优势在于部署简单、数据持久、支持 SQL 查询。缺点是并发写入能力有限，适合单节点或少量节点的部署场景。

Redis 存储实现

Redis 存储是分布式生产环境的推荐选择。Redis 是高性能的内存数据库，支持丰富的数据结构和原子操作，非常适合任务队列场景。

Redis 任务存储使用哈希表存储每个任务的详细信息。键名格式为 task:任务标识，字段包含任务的所有属性。同时维护按状态分类的集合，键名格式为 status:状态值，用于快速按状态查询任务。

Redis 队列存储使用有序集合实现优先级队列。键名固定为 queue:priority，成员是任务标识，分数是优先级数值。出队操作使用 Lua 脚本实现原子性，脚本执行 ZRANGE 获取任务、ZREM 移除任务、返回任务列表三个操作。

Lua 脚本在 Redis 服务端执行，整个操作期间不会被打断，保证了原子性。批量出队脚本一次可以获取多个任务，显著减少网络往返次数。

sequenceDiagram
    participant Node as 任务节点
    participant Redis as Redis服务器
    participant Lua as Lua脚本
    
    Node->>Redis: 调用批量出队脚本
    Redis->>Lua: 执行脚本
    
    Lua->>Redis: ZRANGE queue 0 count-1
    Redis-->>Lua: 返回任务ID列表
    
    loop 每个任务
        Lua->>Redis: SETNX lock:task_id node
        Redis-->>Lua: 锁获取结果
    end
    
    Lua->>Redis: ZREM queue task_ids
    Redis-->>Lua: 删除数量
    
    Lua-->>Redis: 返回已锁定任务ID
    Redis-->>Node: 返回任务ID列表

存储工厂模式

存储工厂封装了存储实例的创建逻辑，根据配置参数自动选择相应的存储后端。

工厂模式的核心价值在于解耦。业务代码只需要依赖存储接口，不需要知道具体使用的是哪种存储实现。当需要更换存储后端时，只需要修改配置，业务代码无需任何改动。

存储配置支持三种类型。内存类型不需要额外参数，SQLite 类型需要指定数据库文件路径，Redis 类型需要指定连接地址。配置对象提供了便捷的工厂方法，如 memory、sqlite、redis，用于快速创建特定类型的配置。

数据清理策略

持久化存储需要定期清理过期数据，防止存储空间无限增长。

任务清理策略基于任务的终端状态和生存时间。成功、失败、取消等终端状态的任务在完成一段时间后可以被清理。清理周期默认每小时执行一次，每次清理最多删除一千条过期任务。

SQLite 存储需要定期执行 VACUUM 命令回收删除操作释放的磁盘空间。Redis 存储依赖键过期机制自动删除，不需要额外的清理操作。

队列清理策略处理因节点故障而长期滞留在队列中的任务。当任务在队列中的等待时间超过阈值时，系统会将其移出队列并标记为超时，触发重试或死信处理。

配置示例

不同场景下的存储配置选择。

开发测试场景使用内存存储，提供最快的响应速度，不需要持久化能力。单机生产场景使用 SQLite 存储，数据持久化到本地文件，部署简单且数据可靠。分布式生产场景使用 Redis 存储，支持多节点共享数据，提供高性能和高可用。

总结

NeoTask 的持久化设计通过存储抽象、多种实现和工厂模式的组合，实现了从单机到分布式的平滑演进。

内存存储适用于开发测试，提供最快的读写速度。SQLite 存储适用于单机生产，无需额外部署且数据可靠。Redis 存储适用于分布式生产，支持高并发和水平扩展。

统一的存储接口保证了核心逻辑的稳定性，无论底层使用哪种存储，上层的任务调度和执行逻辑都不需要修改。这种设计使得系统能够根据业务规模和性能要求灵活选择存储方案，而不需要重构代码。
