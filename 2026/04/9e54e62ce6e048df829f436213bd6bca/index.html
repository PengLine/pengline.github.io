分布式任务调度（NeoTask）架构设计

NeoTask 架构设计

📚 返回首页 | 📖 快速上手 | 📘 API 参考 | ⚙️ 配置说明 | 🐙 GitHub | 🐱 NeoTask 官网

概述

NeoTask 是一个纯 Python 实现的轻量级异步任务调度系统，专为耗时任务（AI 生成、视频处理、数据爬取等）设计。无需部署 Redis、PostgreSQL 等外部服务，安装后即可在任意 Python 项目中直接使用。

设计目标

目标 说明
零依赖部署 纯 Python 实现，内置 SQLite 支持
高易用性 简单的 API 设计，开箱即用
可扩展性 支持内存/SQLite/Redis 多种存储后端
高可用 支持分布式部署、故障恢复
高性能 异步并发处理，支持多 Worker

核心特性

  • 即时任务：优先级队列，立即执行

  • 延时任务：指定延迟时间执行

  • 周期任务：固定间隔重复执行

  • Cron 表达式：复杂时间规则调度

  • 任务编排：DAG 工作流，支持条件分支和并行执行

  • 自动重试：失败任务自动重试

  • 持久化存储：内存/SQLite/Redis 多种后端

  • 事件总线：任务生命周期监听

  • 分布式锁：防止任务重复执行

  • 同步/异步等待：灵活的结果获取方式

核心数据流

即时任务数据流

sequenceDiagram
    participant User
    participant Pool as TaskPool
    participant LM as LifecycleManager
    participant QS as QueueScheduler
    participant PF as Prefetcher
    participant WP as WorkerPool
    participant FM as FutureManager
    participant Storage as Storage

    User->>Pool: submit(data, priority, delay)
    Pool->>LM: create_task()
    LM->>Storage: 保存任务(PENDING)
    LM-->>Pool: 返回 Task 对象
    Pool->>QS: push(task_id, priority, delay)
    Pool->>FM: get(task_id)
    Pool-->>User: 返回 task_id
    
    loop 预取循环
        PF->>QS: pop_batch(prefetch_size)
        QS-->>PF: 返回任务列表
        PF->>PF: 放入本地队列
    end
    
    loop Worker 调度循环
        WP->>PF: get() 从本地队列获取
        alt 本地队列非空
            WP->>LM: start_task(task_id)
            WP->>Storage: 更新状态(RUNNING)
            WP->>WP: 获取分布式锁
            
            alt 任务执行成功
                WP->>WP: executor.execute(data)
                WP->>LM: complete_task(result)
                WP->>FM: complete(task_id, result)
                WP->>Storage: 更新状态(SUCCESS)
                WP->>Storage: 释放锁
            else 任务执行失败
                WP->>LM: fail_task(error)
                WP->>FM: complete(task_id, error)
                WP->>Storage: 更新状态(FAILED)
                alt 未超过重试次数
                    WP->>QS: push(重试入队, delay)
                else 超过重试次数
                    WP->>Storage: 移入死信队列
                end
            end
        end
    end
    
    User->>Pool: wait_for_result(task_id)
    Pool->>FM: wait()
    FM-->>Pool: 返回结果
    Pool-->>User: 任务结果

分布式架构设计

去中心化对等架构

NeoTask 采用去中心化的对等架构，所有节点完全对等，没有主节点或从节点的区分。这种设计带来了几个核心优势。

首先是线性扩展能力。由于没有中心化调度节点，系统吞吐量可以随节点数量线性增长。增加新节点只需要启动进程并连接到共享存储，无需任何协调操作。

其次是故障隔离性。单个节点的故障不会影响其他节点的正常运行，也不会触发复杂的恢复流程。其他节点继续从共享队列中抢占任务，故障节点的任务会在锁超时后被其他节点接管。

第三是部署简便性。节点之间没有依赖关系，不需要配置领导者地址，也不需要维护节点间的连接。这使得容器化部署和弹性伸缩变得异常简单。

抢占式消费机制

抢占式消费是去中心化架构的核心。所有节点从共享队列中竞争任务，处理能力强的节点自然获取更多任务，实现自动的负载均衡。

抢占操作通过 Lua 脚本在 Redis 服务端原子执行。脚本从有序集合中取出指定数量的最小分数任务，为每个任务设置分布式锁，锁的值为节点标识和执行时间戳，最后从队列中移除这些任务并返回。整个过程不会被其他操作打断，保证了任务不会被重复分配。

flowchart LR
    subgraph PreemptScript["Lua脚本执行流程"]
        A[开始] --> B[ZRANGE queue 0 count-1]
        B --> C{还有任务?}
        C -->|是| D[SET lock:task NX EX ttl]
        D --> E{锁成功?}
        E -->|是| F[加入结果列表]
        E -->|否| C
        F --> G[ZREM queue task]
        G --> C
        C -->|否| H[返回结果列表]
        H --> I[结束]
    end

预取机制

预取机制通过批量获取任务减少对 Redis 的频繁访问，显著提高系统吞吐量。

每个节点维护一个本地队列。预取循环检查本地队列大小，当队列大小低于最小阈值时触发预取操作，从 Redis 批量抢占任务放入本地队列。当队列大小高于最大阈值时预取循环暂停，等待队列被消耗至低于阈值。

工作协程从本地队列头部获取任务，始终保持优先级顺序。本地队列为空时工作协程等待预取器补充新任务。

flowchart TB
    subgraph PrefetchLoop["预取循环"]
        CheckLocal[检查本地队列大小] --> Compare{对比阈值}
        
        Compare -->|低于最小阈值| Prefetch[执行批量抢占]
        Compare -->|高于最大阈值| SleepLong[长时间休眠]
        Compare -->|介于之间| SleepShort[短时间休眠]
        
        Prefetch --> AddLocal[放入本地队列]
        AddLocal --> Compare
        
        SleepLong --> Compare
        SleepShort --> Compare
    end
    
    subgraph WorkerLoop["工作循环"]
        GetLocal[从本地队列获取] --> Execute[执行任务]
        Execute --> UpdateStats[更新统计]
        UpdateStats --> GetLocal
    end
    
    PrefetchLoop -->|任务| WorkerLoop

高可用设计

节点故障检测

节点故障检测采用分级超时策略。节点每隔固定时间向 Redis 写入心跳时间戳，监控服务定期扫描所有节点的心跳记录。

短超时用于快速感知可能的问题，默认值为心跳间隔的三倍。宽限期介于短超时和长超时之间，用于过滤网络抖动导致的临时失联。长超时用于确认节点死亡，默认值为六十秒。

这种分级设计在故障检测的及时性和准确性之间取得了平衡。过于敏感的检测会导致不必要的任务回收和告警，过于迟钝的检测则会延长故障恢复时间。

    subgraph Detection["故障检测流程"]

        Check -->|心跳正常| Active[标记活跃]
        Check -->|超过短超时| Suspect[标记可疑]
        Check -->|超过长超时| Dead[标记死亡]
        
        Suspect --> Wait[等待宽限期]
        Wait --> CheckAgain{收到心跳}
        CheckAgain -->|是| Active
        CheckAgain -->|否| Dead
        
        Dead --> Reclaim[回收节点任务]
        Reclaim --> Alert[发送告警]
    end

孤儿任务回收

当节点崩溃且其持有的锁过期后，原本由该节点执行的任务成为孤儿任务。回收协程定期扫描运行中状态的任务，检查关联的分布式锁是否存在。

如果锁不存在或已过期，说明原执行节点可能已崩溃，将该任务标记为孤儿并应用恢复策略。恢复策略根据任务类型、优先级、重试次数决定是立即重试、延迟重试还是移入死信队列。

flowchart TB
    subgraph ReclaimCycle["回收周期"]
        Start([开始]) --> AcquireLock[获取回收锁]
        AcquireLock --> LockSuccess{获取成功}
        
        LockSuccess -->|否| Skip[跳过本轮]
        LockSuccess -->|是| Scan[扫描运行中任务]
        
        Scan --> Loop{还有任务}
        Loop -->|是| CheckLock[检查分布式锁]
        
        CheckLock --> LockExists{锁存在}
        LockExists -->|是| CheckNode{节点心跳正常}
        CheckNode -->|是| SkipTask[跳过此任务]
        CheckNode -->|否| MarkOrphan[标记孤儿]
        
        LockExists -->|否| MarkOrphan
        
        MarkOrphan --> ApplyPolicy[应用恢复策略]
        ApplyPolicy --> UpdateStats[更新统计]
        UpdateStats --> Loop
        
        Loop -->|否| ReleaseLock[释放回收锁]
        ReleaseLock --> End([结束])
        
        Skip --> End
        SkipTask --> Loop
    end

任务重试与死信

任务执行失败时，系统根据重试配置决定处理方式。重试采用指数退避策略，每次重试的延迟时间递增，避免对下游服务造成压力。

超过最大重试次数的任务移入死信队列。死信队列存储失败任务的完整信息，包括原始数据、失败原因、重试历史和时间戳。运维人员可以通过管理工具查看死信详情，决定重新执行、跳过还是修复数据。

flowchart TB
    subgraph RetryFlow["重试流程"]
        Task[任务执行失败] --> CheckRetry{重试次数检查}
        
        CheckRetry -->|未超限| CalculateDelay[计算延迟时间]
        CalculateDelay --> Requeue[重新入队]
        Requeue --> UpdateStats[更新重试计数]
        
        CheckRetry -->|超限| ToDLQ[移入死信队列]
        ToDLQ --> StoreInfo[存储失败信息]
        StoreInfo --> SendAlert[发送告警]
        
        ToDLQ --> Classify{分类处理}
        Classify -->|临时故障| AutoRetry[自动重试]
        Classify -->|业务错误| Manual[人工处理]
        Classify -->|确认无用| Archive[归档清理]
    end

监控与可观测性

指标收集体系

NeoTask 内置了多层次的监控指标收集能力。指标采集器周期性收集任务计数、执行时间、队列长度等数据。系统采集器通过系统调用获取 CPU 使用率、内存占用、线程数等资源指标。事件采集器监听事件总线，记录任务状态变更、节点状态变更等关键事件。

采集到的指标经过聚合后可以导出到多个目的地。控制台输出适合开发和调试场景，REST API 供外部系统集成，Prometheus 端点与主流监控生态对接。

健康检查

健康检查是自动化运维的基础。存活检查通过心跳检测判断节点是否还在运行。就绪检查判断节点是否能够接受新任务，节点启动时需完成组件初始化才能通过就绪检查。深度检查包括存储连接测试、队列操作测试、锁操作测试等，深度检查失败意味着节点功能受损。

存储后端

存储抽象

NeoTask 通过定义统一的存储接口，将任务数据的操作与具体存储实现解耦。任务存储接口负责任务的创建、查询、更新、删除和列表操作。队列存储接口负责队列的入队、出队、大小查询和清空操作。

存储选型建议

不同场景下的存储选型策略有所不同。

开发测试场景使用内存存储，提供最快的响应速度，不需要持久化能力。单机生产场景使用 SQLite 存储，数据持久化到本地文件，部署简单且数据可靠。分布式生产场景使用 Redis 存储，支持多节点共享数据，提供高性能和高可用。
