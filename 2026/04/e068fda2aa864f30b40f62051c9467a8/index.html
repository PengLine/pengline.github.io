NeoClip 技术架构

相关链接：GitHub | 官网 | 文档 （MCP Server · 集成指南）

线性链路节点模式

用户上传视频 → 快速采样 → 用户输入指令 → LLM 解析指令 →【展示规划结果给用户确认】→ 用户修改/确认 → 完整视频分析 → 片段匹配 → 合成

& 快速采样] --> PARSE
    PARSE[指令解析智能体
理解用户意图] --> REVIEW{用户审核规划}
    REVIEW -- 满意 --> ANALYZE
    REVIEW -- 不满意 --> MODIFY[用户直接修改规划]
    MODIFY --> REVIEW

    MATCH -.->|片段不足| MODIFY

智能体节点

信息采样智能体

不进行完整分析，只快速提取理解指令所需的最小信息。比如，从每个视频中间抽 3-5 帧，生成缩略图，供用户和 LLM 参考，了解“视频里大概有什么”。这为指令解析提供视觉上下文，帮助 LLM 更准确地映射术语。

指令解析智能体

  • 输入：用户指令 + 快速采样的缩略图/视觉概要。

  • 核心变化：其产出不再是最终执行脚本，而是一个可视化的规划提案。

  • 输出：可交互的时间线草图，包含每个槽位的语义描述、预估时长，并附带从采样帧中挑出的“可能匹配”预览图。

交互设计要点

用户修改规划时，界面可以参考专业剪辑软件的时间线，但简化其概念，使用故事板视图：

  • 横向时间轴：分槽位显示。

  • 每个槽位卡片上有：语义描述（可编辑文本）、预估时长、来源视频、可选的转场图标（点击可切换）。

  • 槽位之间有明显的“+”号和转场选择器。

用户审核与修改

这是 LangGraph 发挥价值的地方。REVIEW 对应一个 interrupt 断点，流程暂停，把规划提案推给前端。

用户界面展示提案，用户可以进行如下操作：

  • 整体确认：允许继续。

  • 调整顺序：拖拽槽位，把“活动”放到“自我介绍”前面。

  • 修改语义：把“寻找风景片段”改成“寻找室内交谈场景”。

  • 指定来源：明确指出，某一段必须用“视频A”的内容。

  • 增删槽位：加一个纯文字片尾，或删掉不满意的部分。

用户修改后，新的规划作为指令，重新注入状态，唤醒后续智能体。这正是 LangGraph Command 或更新状态的核心机制。

架构优劣势

优势

  1. 尊重用户控制权：用户是导演，AI 是场记和剪辑助理。关键创意决策由用户确认。

  2. 避免无效计算：先规划、后分析，只在用户明确需求后才调用最耗资源的模型。

  3. 优雅处理失败：通过“空缺报告”将机器发现的困难转化为选项，交给用户决策，而不是系统报错退出。

  4. 渐进的复杂度：初级用户可以直接点“确认”，高级用户可以深度调整。

这个“规划-确认-执行”的闭环，让系统从一次性黑盒变为协作工具。这在技术架构上，恰好就是 LangGraph 的 interrupt 和状态持久化所擅长的模式。我们可以在这个共识基础上，继续深入某个智能体的内部逻辑，或者视频分析的具体方案。

劣势

星型中枢分发模式

一种多层次的动态控制

  1. 解耦指令与流程：流水线是固定的，但指令可以随时降临在任何阶段。中枢的存在让新指令的接入不影响原有流水线的逻辑。

  2. 无缝的“宏观-微观”过渡：用户既能说“做个旅行Vlog”（宏观规划），也能在后续说“把这段海浪声加大”（微观操作），两者在同一个架构下和谐统一。

  3. 强大的上下文感知：通过 State 维护活跃对象，用户可以用代词和模糊指代，体验更自然。

  4. 易于扩展：以后想加入“添加字幕”或“颜色校正”等新能力，只需：

    • 增加一个新的智能体节点。
    • 在中枢的意图分类器里增加一个新类别。
    • 配置新的路由规则。核心架构不变。

中枢内部：意图识别与参数提取

这个模块是大脑，它要回答两个问题：“用户想操作什么？” 和 “具体参数是什么？”。

  1. 意图分类：维护一个意图分类体系。例如：

    • PLAN_CREATE：创建新的时间线
    • PLAN_MODIFY：修改时间线结构（增/删/移动槽位）
    • ANALYZE_SET：设置分析优先级
    • CLIP_TRIM：精确修剪片段
    • CLIP_REPLACE：替换某个槽位的片段
    • EFFECT_ADD：添加特效
    • AUDIO_ADJUST：调整音频
    • STATE_QUERY：查询当前状态（“现在进行到哪了？”）

  2. 槽位填充：从指令中提取具体参数。例如“把[第三个片段]和[第二个片段]对调”，提取出 slot_ids。

  3. 上下文消歧：这很关键。用户的“删掉它”需要结合前端 UI 当前选中的对象，中枢需要能访问这部分上下文。LangGraph 的 State 可以持有 active_slot_id 之类的信息来辅助理解。

指令解析中枢

第一级：全局性规划指令 (会话开始时)

这是最宏观的指令，旨在定义最终成片的结构和风格。

  • 用户说法：“把自我介绍放开头，活动放中间，风景放结尾，用温馨的配乐。”

  • 中枢作用：解析并生成第一版完整时间线蓝图。这是后续所有智能体的行动总纲。

第二级：智能体导向的操作指令 (执行过程中)

这是针对特定智能体或当前任务的具体指令，也就是你提到的“分析指令”、“剪切指令”。

  • 用户说法：“优先分析视频A”、“把这个片段剪到10秒以内”、“这里的转场换成闪白”、“背景音乐声音小一点”。

  • 中枢作用：解析并识别意图类型（分析、剪辑、特效、音频……），然后路由到对应的智能体。它不修改时间线蓝图，而是精确地给某个智能体下达一个原子任务。

第三级：属性修改指令 (任何时候)

这是最轻量的指令，直接修改时间线蓝图或某个片段的属性，不一定需要智能体介入。

  • 用户说法：“把‘温馨’改成‘活力’”、“把第三个片段和第二个片段对调一下”。

  • 中枢作用：直接解析并更新 VideoAssemblyState 中的对应字段（如 global_context.mood 或 timeline_slots 的顺序），然后系统根据新状态自动决定下一步。

中枢与智能体的关系

中枢和智能体之间，应该是 松耦合的命令分发关系，而不是父子控制关系。

1.1 设计原则：基于能力的注册与发现

  • 智能体只暴露能力，不暴露存在：每个智能体向中枢注册的是“我能做什么”（如 CLIP_TRIM、EFFECT_ADD），而不是“我叫什么名字”。

  • 中枢只关心意图，不关心实现：中枢识别出用户意图后，查找能处理该意图的智能体，然后分发命令。

  • 智能体可插拔：以后要加“字幕智能体”，只需它注册 SUBTITLE_ADD 能力，中枢完全不用改。

中枢与用户的交互

多模态、上下文感知、反馈闭环

2.1 交互模式：三种基本模式

模式 说明 示例
一次性指令 用户一句话说清楚，中枢直接执行 “把第一个视频的自我介绍片段剪出来”
澄清式对话 指令模糊，中枢反问，用户确认 用户：“把那段剪掉”。中枢：“您指的是刚预览的’海边散步’片段吗？”
渐进式细化 先粗后细，中枢记住上下文 用户：“做个Vlog。”…完成后，用户：“把中间那段调亮一点。”中枢知道“中间那段”指什么

第三种是体验分水岭，依赖中枢的上下文记忆。

2.2 上下文记忆机制

中枢需要维护一个对话上下文对象，它不同于 VideoAssemblyState（那是全局生产状态），这是交互层上下文：

python

当用户说“把它调亮一点”，中枢会：

  1. 检查 active_slot_id，确定“它”是哪个槽位。

  2. 检查 last_previewed_clip，如果 active_slot_id 为空，就用这个。

  3. 如果两者都为空，进入澄清模式，反问用户。

这个上下文在 LangGraph 的 State 中作为子对象存储，每一步都持久化到 checkpointer。

2.3 澄清与确认的边界

不是所有指令都要确认。中枢需要判断风险等级：

  • 低风险（可撤销操作）：调音量、换转场、预览片段 → 直接执行，允许撤销。

  • 中风险（影响局部结果）：替换某段素材、裁剪某个片段 → 执行并高亮展示变化，用户可回退。

  • 高风险（影响全局结构）：重新规划时间线、删除整个槽位、开始最终渲染 → 必须先确认再执行。

中枢在路由命令前，先过一遍风险评估，决定是直接执行还是触发 interrupt 等待确认。
