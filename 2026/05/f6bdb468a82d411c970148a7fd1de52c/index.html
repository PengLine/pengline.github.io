剧本分镜智能体（PenShot）向量检索

发表于2026-05-21|更新于2026-05-21

PenShot 向量检索

相关链接：GitHub | 官网 | 文档 （架构设计 · 召回策略 · MCP Server · 一致性策略 · RAG 增强）

概述

向量检索是 PenShot 剧本分镜智能体的核心能力之一，它使系统能够“记住”过去处理过的剧本片段、成功生成的提示词模板以及常见的修复模式。通过将文本内容转换为高维向量空间中的点，系统可以快速找到与当前查询最相似的历史记录，从而实现经验驱动的智能决策。

在 PenShot 的工作流中，向量检索主要服务于以下场景：质量审查时查找相似的历史问题及其解决方案、提示词生成时检索成功的模板进行参考、连续性检查时寻找相似的场景过渡模式。这种基于语义相似度的检索方式，相比传统的关键词匹配，能够更好地理解用户的自然语言查询意图。

向量检索原理

从文本到向量

向量检索的核心思想是将非结构化的文本数据转换为结构化的数学向量。这个过程由嵌入模型完成——一个经过大规模语料训练的深度学习模型，能够将任意长度的文本映射为固定维度的浮点数向量。

    subgraph Input["输入层"]
        T1["剧本文本"]
        T2["提示词模板"]
        T3["角色描述"]
    end
    
    subgraph Embedding["嵌入模型"]
        E["Embedding Model\n(768/1024/1536维)"]
    end
    
    subgraph Vector["向量空间"]
        V1["v1: [0.12, -0.34, 0.56, ...]"]
        V2["v2: [0.45, -0.12, 0.78, ...]"]
        V3["v3: [-0.23, 0.67, -0.45, ...]"]
    end
    
    T1 --> E
    T2 --> E
    T3 --> E
    E --> V1
    E --> V2
    E --> V3

在这个高维向量空间中，语义相似的文本会被映射到彼此接近的位置。例如，“林然裹着旧羊毛毯”和“年轻女性裹着羊毛毯”这两个描述虽然文字不完全相同，但它们的向量在空间中是相近的。这种性质使得向量检索能够理解同义词、近义词和上下文语义。

余弦相似度计算

检索时，系统将用户的查询文本也转换为向量，然后计算查询向量与库中每个文档向量的余弦相似度。余弦相似度的取值范围在 -1 到 1 之间，值越接近 1 表示两个向量方向越一致，语义越相似。

    QV["查询向量\n[q1, q2, q3, ...]"]
    
    subgraph Documents["文档库"]

        R2["2. 场景描述 (0.44)"]

检索流程架构

PenShot 的向量检索系统采用了分层架构，从文档加载到索引创建再到查询检索，每个环节都有明确的职责划分。

    subgraph DataLayer["数据层"]
        D1["剧本解析结果\n(场景/角色/动作)"]
        D2["成功提示词模板\n(历史高评分模板)"]
        D3["修复案例\n(问题+解决方案)"]
    end
    
    subgraph IndexLayer["索引层"]
        I1["文本分块\n(SentenceSplitter)"]
        I2["向量化\n(Embedding Model)"]
        I3["存储\n(Chroma/SimpleVectorStore)"]
        I4["索引管理\n(按script_id隔离)"]
    end
    
    subgraph RetrievalLayer["检索层"]
        R1["查询向量化"]
        R2["余弦相似度计算"]
        R3["重排序\n(BGE-reranker)"]
        R4["结果过滤与格式化"]
    end
    
    subgraph Application["应用层"]
        A1["提示词增强\n检索相似模板"]
        A2["质量审查\n查找历史问题"]
        A3["连续性检查\n匹配过渡模式"]

    I1 --> I2 --> I3 --> I4

核心用法

初始化知识库

在使用向量检索功能之前，需要先初始化剧本知识库实例。知识库会为每个剧本创建独立的存储目录，实现数据隔离。

添加剧本到知识库

知识库支持三种添加方式：直接添加已解析的剧本对象、添加剧本文本、或添加剧本文件。

添加过程中，系统会自动完成文本分块、向量化生成和持久化存储。每个文档节点都会携带元数据，包括类型（场景/角色）、来源剧本ID等，便于后续过滤。

执行向量检索

查询知识库时，系统会将查询文本转换为向量，然后计算与库中所有文档的相似度，最后返回最匹配的结果。

查询返回的结果按相似度从高到低排序，每个结果包含相似度分数、文档原文和元数据信息。

结构化查询

除了语义检索，知识库还支持基于元数据的结构化查询，可以按场景ID或角色名精确查找。

应用场景

提示词生成增强

在生成 AI 视频提示词时，系统会检索历史上成功的高质量提示词模板，作为参考来优化当前片段的提示词。这种经验驱动的生成方式能显著提升提示词的质量和一致性。

    end
    
    F --> R
    T1 --> R
    T2 --> R
    R --> Enhancement
    Enhancement --> P

质量审查辅助

质量审查阶段，系统会检索历史上出现过的问题模式及其修复方案。如果当前生成的内容与某个历史问题相似，系统可以自动应用相应的修复策略。

    A["质量审查发现\n提示词过短"] --> B["向量检索\n相似问题模式"]
    B --> C{找到相似案例?}
    C -->|是| D["应用历史修复方案\n扩展提示词描述"]

    D --> F["生成修复后的\n高质量提示词"]
    E --> F

连续性一致性检查

在跨片段连续性检查中，系统可以检索之前片段的角色状态、场景布置等信息，确保当前片段与历史片段保持视觉和叙事上的一致性。

数据存储结构

PenShot 的向量数据采用按剧本隔离的目录结构存储，每个剧本拥有独立的存储空间，避免了数据混淆。

flowchart TB
    subgraph Root["data/embedding/"]
        subgraph ScriptKB["script_kb/"]
            S1["script_001/"]
            S2["script_002/"]
            S3[".../"]
        end
        
        subgraph PromptTemplates["prompt_templates/"]
            P1["script_001/"]
            P2["script_002/"]
            P3[".../"]
        end
    end
    
    subgraph ScriptDir["script_001/ 目录内容"]
        V["vector_store.json\n(向量数据)"]
        D["docstore.json\n(文档节点)"]
        I["index_store.json\n(索引元数据)"]
        G["graph_store.json\n(关系图)"]
        R["parsed_results/\n(解析结果缓存)"]
    end
    
    S1 --> ScriptDir

这种目录结构设计有以下优势：每个剧本的数据相互隔离，删除剧本只需删除对应目录；支持增量备份和恢复；便于调试和问题排查。

重排序优化

基础向量检索返回的结果可能包含一些语义相关但实际不太适合的结果。PenShot 集成了 BGE-reranker 重排序模型，在初次检索后对结果进行二次精排，提高检索精度。

flowchart LR
    Q["用户查询"]
    V["向量检索\n(返回 Top-20)"]
    R["重排序模型\n(Cross-Encoder)"]
    F["最终结果\n(Top-5)"]
    
    subgraph Scores["相似度分数"]
        S1["初筛: 0.85, 0.72, 0.68,..."]
        S2["重排后: 0.92, 0.78, 0.71,..."]
    end
    
    Q --> V --> S1
    S1 --> R
    R --> S2
    S2 --> F

向量检索（双编码器）和重排序（交叉编码器）的区别在于：

  • 双编码器将查询和文档分别编码，速度快但精度相对较低；
  • 交叉编码器将查询和文档拼接后一起编码，精度高但速度慢。

两者结合可以兼顾效率和精度。

注意事项

  • 嵌入模型一致性

    创建索引时使用的嵌入模型必须与查询时使用的模型完全相同，否则向量空间不一致，相似度计算将失去意义。PenShot 会自动处理这个问题，但如果你手动管理存储文件，需要特别注意。

  • 相似度阈值设置

    LlamaIndex 检索器默认内置了约 0.7 的相似度阈值，这会过滤掉大量有效结果。建议采用手动计算余弦相似度的方式，完全控制阈值逻辑。

  • 存储文件完整性

    向量存储需要同时保存 vector_store（向量数据）和 docstore（文档节点），两者缺一不可。使用 StorageContext.from_defaults(persist_dir=...) 可以同时加载所有相关文件，避免文件不一致的问题。

  • 内存与性能考虑

    向量检索需要将索引加载到内存中，对于大规模的剧本库（数万文档），建议使用 Chroma 等专业向量数据库替代 SimpleVectorStore。PenShot 通过按剧本隔离存储，限制了单次加载的数据量。

  • 多剧本隔离

    始终在查询时指定 script_id 参数，确保只检索当前剧本的数据。如果忘记指定，可能检索到其他剧本的内容，导致上下文混乱。

问题排查与解决

问题现象

在使用 LlamaIndex 构建的剧本知识库中，向量检索始终返回 0 条结果，尽管：

  • 向量存储文件存在且完整（3个向量，768/1024维）

  • 文档存储文件存在且完整（3个文档节点）

  • 手动计算的余弦相似度正常（0.44-0.64）

根本原因

  • 原因一：LlamaIndex 检索器内置相似度阈值过高

    index.as_retriever() 方法默认内置了相似度阈值（约为 0.7），所有相似度低于该阈值的结果都会被自动过滤。

    即使不显式设置 similarity_threshold 参数，检索器内部仍然有默认过滤机制。

  • 原因二：vector_store.query() 查询模式不匹配

    vector_store.query() 方法在默认查询模式下无法正确返回结果，可能与 LlamaIndex 版本或配置有关。

  • 原因三：索引加载方式错误

    使用 VectorStoreIndex.from_vector_store() 加载已有存储时，需要同时包含 vector_store 和 docstore，但该方法无法正确处理已持久化的存储上下文。

解决方案

  • 方案一：正确加载索引（修复加载错误）

  • 方案二：手动计算余弦相似度（绕过检索器阈值）

  • 方案三：完整修复代码

关键经验总结

问题 错误做法 正确做法
索引加载 VectorStoreIndex.from_vector_store() StorageContext.from_defaults() + VectorStoreIndex()
向量检索 index.as_retriever().retrieve() 手动计算余弦相似度
相似度过滤 依赖检索器默认阈值 手动计算，按需过滤
查询向量获取 aget_query_embedding() (异步) embed_query() (同步)
