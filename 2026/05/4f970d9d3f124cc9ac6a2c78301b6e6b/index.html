NeoTask 故障恢复

📚 返回首页 | 📖 快速上手 | 📘 API 参考 | ⚙️ 配置说明 | 🐙 GitHub | 🐱 NeoTask 官网

概述

在分布式系统中，节点故障是常态而非异常。网络中断、进程崩溃、机器宕机、资源耗尽，故障的形式多种多样，但系统对故障的响应必须是一致的、可靠的。

NeoTask 的故障恢复机制建立在去中心化架构之上，利用分布式锁、心跳检测、状态持久化等基础能力，实现了自动化的任务恢复。本章将详细阐述故障检测、任务回收、幂等性保证、死信队列等核心设计。

故障恢复架构

故障恢复架构由检测层、决策层和执行层三个层次组成。

检测层负责发现故障。节点心跳监控器定期扫描所有注册节点的心跳时间戳，识别可疑节点和死亡节点。锁过期扫描器检查分布式锁的状态，发现因节点崩溃而过期的锁。任务状态扫描器检查处于运行中状态的任务，识别那些长期未完成的可疑任务。

决策层负责判断如何处理故障。恢复策略引擎根据任务类型、优先级、重试次数、时间窗口等因素，决定是立即重试、延迟重试、跳过还是移入死信队列。

执行层负责具体操作。任务回收器将孤儿任务重新入队或移交。死信处理器将无法自动恢复的任务存入死信队列并触发告警。状态恢复器清理故障节点留下的脏数据。

故障检测流程

故障检测是故障恢复的第一步。NeoTask 采用多层次的检测机制，确保能够及时、准确地发现各种类型的故障。

节点级别的故障检测通过心跳机制完成。每个节点每隔固定时间向 Redis 写入心跳时间戳。检测协程定期扫描所有节点的心跳记录，计算当前时间与最后心跳时间的差值。根据差值大小，将节点标记为活跃、可疑或死亡。

任务级别的故障检测通过锁状态和任务状态共同判断。当节点的锁因 TTL 过期而自动释放，但对应的任务状态仍为运行中时，该任务被识别为孤儿任务。孤儿任务表明原执行节点可能已经崩溃，需要被回收。

任务状态机

任务状态机是故障恢复的核心模型。每个任务在其生命周期中经历多个状态，状态转换由节点操作和故障检测共同驱动。

任务创建后进入创建状态。如果任务有延迟要求，进入调度状态等待执行时间到达。执行时间到达后进入排队状态，等待被节点抢占。节点抢占成功后进入已分配状态，此时任务已被锁定但尚未开始执行。任务开始执行后进入运行中状态。

正常完成的任务进入成功状态，执行失败的任务进入失败状态。成功和失败都是终态，但失败状态可能触发重试。重试时任务重新进入排队状态。

当节点在任务执行过程中崩溃时，任务可能从已分配或运行中状态进入孤儿状态。恢复机制将孤儿任务重新放入排队状态或移入死信队列。

孤儿任务回收

孤儿任务回收是故障恢复的核心功能。当节点崩溃且其持有的锁过期后，原本由该节点执行的任务成为孤儿任务，需要被其他节点接管。

回收流程由回收协程定期触发。每个回收周期执行以下步骤：扫描所有运行中状态的任务，对每个运行中任务检查其关联的分布式锁。如果锁不存在或已过期，说明原执行节点可能已崩溃，将该任务标记为孤儿。对孤儿任务应用恢复策略，根据任务类型、优先级、重试次数等因素决定处理方式。将决定重试的任务重新放入队列，将决定放弃的任务移入死信队列。

为了防止多个节点同时回收同一个任务，回收操作本身也使用分布式锁保护。只有成功获取回收锁的节点才能执行回收，其他节点跳过本轮回收。

flowchart TB
    subgraph ReclaimCycle["回收周期"]
        Start([开始]) --> AcquireLock[获取回收锁]
        AcquireLock --> LockSuccess{获取成功}
        LockSuccess -->|否| Skip[跳过本轮]
        LockSuccess -->|是| Scan[扫描运行中任务]
        
        Scan --> Loop{还有任务}
        Loop -->|是| CheckLock[检查分布式锁]
        CheckLock --> LockExists{锁存在}
        
        LockExists -->|是| CheckNode{节点心跳正常}
        CheckNode -->|是| SkipTask[跳过此任务]
        CheckNode -->|否| MarkOrphan[标记孤儿]
        
        LockExists -->|否| MarkOrphan
        
        MarkOrphan --> ApplyPolicy[应用恢复策略]
        ApplyPolicy --> UpdateStats[更新统计]
        UpdateStats --> Loop
        
        Loop -->|否| ReleaseLock[释放回收锁]
        ReleaseLock --> End([结束])
        
        Skip --> End
        SkipTask --> Loop
    end

恢复策略矩阵

不同的任务需要不同的恢复策略。NeoTask 根据任务的多维属性动态选择恢复策略。

任务优先级是最重要的决策因素。关键优先级任务要求立即重试，恢复时间目标在十秒以内。高优先级任务要求快速恢复，三十秒内重新入队。普通优先级任务采用标准恢复流程，一至两分钟内重试。低优先级任务可以接受较长的恢复延迟。

任务类型也影响策略选择。普通即时任务适合立即重试，重试延迟采用指数退避。延迟任务需要重新计算剩余延迟时间，确保不会过早执行。周期任务需要检查错过的时间窗口，决定是补执行还是跳过本次。

重试次数是终止条件。未超过最大重试次数的任务重新入队，超过最大重试次数的任务进入死信队列。

幂等性保证

任务不能丢失的要求意味着任务可能会被执行多次。故障恢复中的重试机制必然导致部分任务被重复执行，因此幂等性设计至关重要。

NeoTask 通过多层机制保证幂等性。首先是任务标识的全局唯一性，每个任务都拥有全局唯一的标识，这个标识可以包含业务语义，便于业务层实现幂等检查。其次是执行结果的先写后读，任务执行前先检查该标识是否已有成功的执行记录，如有则直接返回缓存结果，跳过实际执行。第三是状态机的严格转换，只有处于允许状态的任务才能被执行，避免重复进入执行流程。

业务层也需要配合实现幂等性。例如，对于订单处理任务，可以根据订单号检查订单状态，已经处理完成的订单直接返回成功。对于数据同步任务，可以使用版本号或时间戳判断是否需要更新。

    subgraph IdempotentCheck["幂等检查流程"]
        Start([收到任务]) --> CheckRecord[查询执行记录]

        Execute --> CheckResult{执行结果}

        CheckResult -->|失败| WriteFailure[写入失败记录]

死信队列设计

死信队列是故障恢复的最后一道防线。当任务经过多次重试仍然失败时，系统不再无限重试，而是将任务移入死信队列等待人工处理。

死信队列存储失败任务的完整信息，包括任务原始数据、失败原因、重试历史、时间戳等。这些信息为问题排查和任务恢复提供依据。

死信队列支持多种处理方式。自动重试适用于临时性故障，例如网络闪断、数据库连接超时等，可以设置延迟后重新入队。人工处理适用于业务逻辑错误或需要人工介入的场景，运维人员可以通过管理工具查看死信详情，决定是重新执行、跳过还是修复数据。归档清理适用于确认无用的死信，定期清理以释放存储空间。

恢复时间保证

NeoTask 承诺在五分钟内完成故障恢复。这个承诺基于对各环节时间上限的严格控制和分析。

心跳检测采用两秒间隔，六秒无心跳触发可疑标记。宽限期设为三十秒，过滤临时性网络故障。确认死亡的最长时间为六十秒。

故障确认后，回收扫描器每十秒执行一次。在最坏情况下，任务在锁过期后等待扫描周期才能被回收，这个等待时间不超过十秒。

恢复决策和重新入队操作在毫秒级别完成。任务重新入队后，其他节点在抢占循环中很快就会获取并执行。

综合各环节的最长时间，故障发现最长六十秒，扫描等待最长十秒，重新执行前的总时间约七十秒。考虑网络延迟和处理开销，整体恢复时间仍在一百二十秒以内，远低于五分钟的目标。

配置参数

故障恢复相关的配置参数需要根据实际环境进行调整。心跳间隔控制故障检测的敏感度，较短的间隔能够更快发现故障，但会增加系统开销。锁超时时间影响任务被其他节点接管的速度，过短可能导致正常执行的任务被误回收，过长则延长恢复时间。重试次数和重试延迟需要在快速恢复和避免无限重试之间取得平衡。

Redis 连接配置影响存储层的可用性，建议使用连接池并配置合理的超时时间。扫描器频率需要与任务量和节点规模匹配，过于频繁会增加系统负载，过低则延迟故障发现。

总结

NeoTask 的故障恢复机制建立在对分布式系统本质规律的深刻理解之上。节点故障是常态而非异常，系统必须能够自动、可靠地处理各种故障场景。

通过对等架构下的心跳检测、分布式锁、任务状态机、恢复策略矩阵等机制的组合，NeoTask 能够在各种故障场景下保证任务的最终执行。五分钟的恢复时间承诺经过严格的流程分析和时间测算，能够满足绝大多数业务场景的需求。

死信队列和告警机制为无法自动恢复的任务提供了兜底方案，确保问题可追溯、可处理。幂等性设计则从根本上解决了重复执行可能带来的数据一致性问题。
