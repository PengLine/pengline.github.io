PenShot 召回评测报告

相关链接：GitHub | 官网 | 文档 （架构设计 · 召回策略 · MCP Server · 一致性策略 · RAG 增强）

测试环境

测试在一台配备16GB内存的机器上进行，Chroma向量数据库存储了来自50个历史任务的约2000个文档。嵌入模型使用BAAI/bge-small-en-v1.5，向量维度768。重排序模型使用BAAI/bge-reranker-base。

2000条文档)]

768维]
        Rerank[BGE-reranker-base]
    end

评测概述

召回系统是PenShot智能增强层的核心组件，负责从历史成功案例和知识库中检索相关信息，以增强当前任务的处理质量。评测围绕三个核心能力展开：向量检索的准确性和效率、重排序对检索结果的优化效果、以及三层记忆系统在不同查询场景下的表现。

        Router --> S[结构化查询]
        
        V --> Fusion[结果融合]

        S --> Fusion

    V -.-> VDB

评测数据集包含来自多个已完成任务的成果，涵盖都市情感、悬疑推理、科幻冒险等多种剧本类型。这些数据被索引为数千个向量文档，存储于Chroma向量数据库中。检索查询来自真实的分镜任务，涉及角色一致性验证、场景连续性检查、提示词模板匹配等典型场景。

向量检索性能

相似度检索

标准相似度检索使用余弦相似度计算查询向量与文档向量的距离。在大量检索测试中，平均检索耗时为毫秒级，召回前多个结果的平均准确率达到了较高水平。

flowchart LR
    subgraph Query["查询处理"]
        QT[查询文本] --> Embed[嵌入模型]
        Embed --> QV[查询向量]
    end
    
    subgraph Index["向量索引"]
        direction TB
        V1[(文档向量1)]
        V2[(文档向量2)]
        V3[(文档向量...)]
        VN[(文档向量N)]
    end
    
    subgraph Search["相似度计算"]
        QV --> Cosine[余弦相似度]
        Cosine --> V1
        Cosine --> V2
        Cosine --> V3
        Cosine --> VN
    end
    
    subgraph Result["结果排序"]
        Sorted[按分数排序] --> Top[Top-K结果]
    end

检索延迟分布显示，绝大多数查询在100毫秒内完成，极少数复杂查询耗时略长。延迟主要受索引大小和向量维度影响，较高的嵌入维度在大规模索引下仍能保持亚秒级响应。

召回准确率在不同查询类型间存在差异。角色描述查询的准确率最高，因为角色名称和特征具有较高的辨识度。场景氛围查询的准确率次之。动作序列查询的准确率相对较低，因为动作描述的语义多样性更高，难以精确匹配。

MMR检索

MMR检索在相似度基础上增加了多样性惩罚因子，避免返回语义冗余的结果。当用户查询需要获取多种解决方案时，MMR模式的性能优势明显。

graph LR
    subgraph MMR["MMR检索原理"]
        R[候选结果集] --> Score[相似度计算]
        Score --> Div[多样性计算]
        Div --> Combine[加权组合]
        Combine --> Select[选择最优]
        Select --> Update[更新已选集]
        Update -.->|循环| Score
    end
    
    subgraph Formula["计算公式"]
        F["MMR = argmax[ λ·Sim(Q,D) - (1-λ)·max Sim(D, D_j) ]"]
    end

测试中设置了合理的相似度权重和多样性权重。MMR返回的前几个结果之间的平均语义相似度明显低于标准相似度检索，多样性提升显著，同时召回准确率仅有可接受的下降。

具体案例中，查询提示词模板时，标准检索返回的结果都来自同一类情感场景，MMR则返回了分别对应不同情绪类型的处理方式。这种多样性对于提示词融合策略尤其有价值，可以避免生成提示词的风格固化。

重排序优化效果

重排序使用专门的模型对初步检索结果进行精细排序。该模型基于交叉编码器架构，能够捕捉查询与文档之间的深层语义交互，效果显著优于双编码器的向量相似度。

flowchart TB
    subgraph TwoStage["两阶段检索策略"]
        subgraph Stage1["第一阶段: 向量检索"]
            Q1[查询] --> E1[嵌入]
            E1 --> S1[相似度计算]
            S1 --> R1[Top-20候选]
        end
        
        subgraph Stage2["第二阶段: 重排序"]
            R1 --> P[构建查询-文档对]
            P --> CE[交叉编码器]
            CE --> S2[精细相关性分]
            S2 --> R2[Top-3结果]
        end
    end

实验数据表明，重排序显著提升了top-K准确率。重排序对边界案例的改进最为明显，那些与查询语义相近但向量距离较远的结果被成功推到了前列。

重排序也带来了额外的计算开销。单次重排序耗时是向量检索耗时的数倍。为了平衡效率与效果，系统采用了先检索后重排序的策略，先用向量检索获取候选结果，再用重排序模型精选最终结果。这种两阶段策略将计算开销控制在了可接受范围。

重排序模型对查询长度也有一定的敏感性。短查询时重排序效果提升有限，因为缺少足够的上下文进行精细对比。长查询时效果最好，丰富的语义信息让模型能够更准确地区分相关与不相关结果。

三层记忆系统评测

短期记忆

短期记忆基于Redis实现，配置了合理的消息滑动窗口和TTL。在测试场景中，系统连续处理多个查询后，短期记忆窗口正确维护了最新的交互记录，旧记录被自动淘汰。

stateDiagram-v2
    [*] --> 空闲
    
    空闲 --> 写入: 添加记忆
    写入 --> 活跃: 存储成功
    
    活跃 --> 读取: 检索记忆
    读取 --> 活跃: 返回结果
    
    活跃 --> 淘汰: 超过窗口大小
    淘汰 --> 活跃: 移除旧记录
    
    活跃 --> 过期: TTL超时
    过期 --> 空闲: 自动清理
    
    活跃 --> 清空: 任务完成
    清空 --> 空闲: 释放资源

短期记忆的读写延迟极低，非常适合作为工作流中的即时上下文缓存。Redis连接池配置了合理的连接数，在并发请求下仍能保持稳定的响应时间。

短期记忆的内容在任务完成后被正确清空，避免了不同任务间的数据串扰。这种隔离机制通过为每个任务分配独立的session_id实现。

中期记忆

中期记忆使用摘要压缩机制，将较长的对话历史浓缩为简洁的摘要，存储于JSON文件中。摘要生成时限制了最大tokens，确保摘要不会过度膨胀。

flowchart LR
    subgraph Input
        M1[消息1]
        M2[消息2]
        M3[消息...]
        MN[消息N]
    end
    
    subgraph Compression["摘要压缩"]
        direction TB
        C[合并消息] --> S[调用LLM生成摘要]
        S --> V[验证摘要质量]
        V --> T{长度检查}
        T -->|超限| C
        T -->|通过| O[输出摘要]
    end
    
    subgraph Output
        SUM[结构化摘要
JSON格式]
    end
    
    Input --> Compression --> Output

测试中，处理大量消息的长对话后，原始对话文本显著压缩，生成的摘要保留了核心信息，包括主要的问题类型、已采取的修复措施、以及成功与失败的模式统计。

摘要的生成质量随对话长度增加有所下降。当原始消息过多时，摘要开始丢失部分细节，但关键决策点仍然得到了保留。这一现象提示需要更智能的摘要策略，如分层摘要或主题聚类。

中期记忆的持久化采用增量更新策略。每次阶段完成后，系统异步保存摘要文件，避免了阻塞主流程。文件读写耗时很短，对整体性能影响很小。

长期记忆

长期记忆基于Chroma向量数据库，存储了历史任务的完整成果和问题模式。评测时数据库中包含来自大量历史剧本的向量文档。

graph TB
    subgraph Storage["长期记忆存储"]
        direction LR
        S1[剧本1] --> V1[(向量索引)]
        S2[剧本2] --> V2[(向量索引)]
        S3[剧本...] --> V3[(向量索引)]
        SN[剧本N] --> VN[(向量索引)]
    end
    
    subgraph Retrieval["检索流程"]
        Q[查询] --> E[嵌入]
        E --> Search[相似度搜索]
        Search --> Filter[结果过滤]
        Filter --> Output[相关记忆]
    end
    
    subgraph Metadata["元数据标签"]
        T1[剧本ID]
        T2[场景类型]
        T3[角色名称]
        T4[问题类别]
    end
    
    V1 -.-> Metadata
    V2 -.-> Metadata

检索测试中，针对角色一致性查询，长期记忆成功召回了相似角色在历史场景中的服装描述和位置轨迹。这些信息被用于验证当前任务的连续性约束，减少了跨片段的一致性错误。

针对常见问题的检索，长期记忆能够返回历史上类似问题的解决方案。例如对于镜头时长超限问题，历史记录中包含了多种分割策略，系统根据当前场景的节奏特征选择了最匹配的方案。

长期记忆的更新机制采用了批量索引策略。每当多个新任务完成时，系统触发一次批量索引更新，将成功提示词和质量报告向量化后写入数据库。批量操作减少了对数据库的写入压力。

混合检索策略

混合检索结合了多种检索来源，融合时采用加权合并策略，合理分配不同来源的权重。

flowchart LR
    subgraph Sources["检索来源"]
        V[向量检索] --> W1[权重0.6]
        M[中期记忆] --> W2[权重0.4]
        S[结构化查询] --> W3[权重0.3]
    end
    
    subgraph Fusion["融合策略"]
        N1[分数归一化]
        D[去重处理]
        R[重排序]
    end
    
    subgraph Output
        Final[融合结果]
    end
    
    W1 --> Fusion
    W2 --> Fusion
    W3 --> Fusion
    Fusion --> Output

在提示词生成场景中，混合检索同时从长期知识库获取成功模板，从中期记忆获取当前任务的修复历史，生成的最佳提示词质量评分比单一来源更高。模板融合过程还考虑了历史修复的成功率，倾向于选择那些在类似问题中证明有效的模式。

混合检索的查询延迟是多种来源的最大值加上融合计算开销。融合计算包括分数归一化、去重和重排序，开销很小。

缓存命中率分析

系统对高频查询结果进行了缓存，缓存TTL设置为合理值。在连续处理同一系列的短剧任务时，缓存命中率较高。

flowchart TD
    Q[查询请求] --> Cache{缓存检查}
    Cache -->|命中| Return[直接返回结果]
    Cache -->|未命中| Search[执行检索]
    Search --> Store[存入缓存]
    Store --> Return
    
    subgraph Policy["缓存策略"]
        TTL[TTL过期清理]
        LRU[LRU淘汰机制]
        Limit[容量上限限制]
    end
    
    Store -.-> Policy

命中最高的查询类型是角色描述和场景氛围，因为同一系列剧集中的角色和场景会反复出现。而动作序列和情绪变化的查询命中率较低，因为每个任务的动作编排差异较大。

缓存淘汰采用LRU策略，当缓存达到上限时，最久未使用的结果被淘汰。淘汰算法的命中率与工作负载密切相关，在处理高度重复的批量任务时表现最优。

召回质量对最终输出的影响

召回系统的质量直接决定了提示词增强和连续性验证的效果。评测数据显示，当检索结果的top-1相关性得分较高时，生成的提示词质量评分明显高于无检索增强的情况。

graph LR
    subgraph Impact["召回质量影响链"]
        R[召回质量] --> E[提示词增强]
        E --> P[提示词质量]
        P --> V[视频生成效果]
    end
    
    subgraph Metrics["评估指标"]
        M1[相关性得分]
        M2[多样性得分]
        M3[新颖性得分]
    end
    
    R --> Metrics

在连续性检查中，检索到的历史一致性模式帮助系统识别出了角色服饰跨场景变化的潜在风险，提前发出警告并触发自动修复。这一机制将连续性问题的检出率显著提升。

召回失败时的降级策略也经过了测试。当检索结果为空或所有结果得分低于阈值时，系统回退到基于规则的默认处理。降级场景下的提示词质量评分有所下降，但系统仍能完成基本的分镜功能，不至于完全失败。
