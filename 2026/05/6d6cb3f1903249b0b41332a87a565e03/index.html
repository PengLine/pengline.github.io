NeoTask 分布式锁

📚 返回首页 | 📖 快速上手 | 📘 API 参考 | ⚙️ 配置说明 | 🐙 GitHub | 🐱 NeoTask 官网

概述

在分布式任务调度系统中，多个节点同时处理任务时，必须有一种机制防止同一个任务被多个节点重复执行。分布式锁正是解决这个问题的核心组件。

NeoTask 的分布式锁实现基于 Redis，利用其单线程模型和原子操作能力，提供了高性能、高可用的锁服务。与传统分布式锁不同，NeoTask 的锁设计充分考虑了任务调度的特定场景，加入了看门狗自动续期、批量锁操作、锁扫描与清理等高级特性。

分布式锁在 NeoTask 中扮演着多重角色。任务执行前需要获取锁，防止重复执行。节点抢占任务时通过锁原子地完成获取和锁定。故障恢复时通过锁状态判断任务是否孤儿。可以说，分布式锁是分布式协调的基石。

分布式锁的核心原理

Redis 锁的实现基础

Redis 实现分布式锁的核心命令是 SET NX EX。SET 命令的 NX 参数确保只有当键不存在时才设置成功，EX 参数为键设置过期时间。这两个参数的组合保证了锁的互斥性和自动释放能力。

NX 参数实现互斥性。多个客户端同时尝试设置同一个键，只有一个能够成功，成功的客户端获得锁，失败的客户端知道锁已被占用。

EX 参数实现自动释放。即使获得锁的客户端崩溃，锁也会在过期时间后自动释放，避免死锁。过期时间称为锁的 TTL（生存时间）。

SET NX EX 是原子操作，不存在检查键存在和设置键之间的竞争窗口。这是 Redis 分布式锁可靠性的保证。

flowchart LR
    subgraph SetNxEx["SET NX EX 原理"]
        Client1[客户端 A] -->|SET lock owner A NX EX 30| Redis[(Redis)]
        Client2[客户端 B] -->|SET lock owner B NX EX 30| Redis
        
        Redis --> Result1[成功 获得锁]
        Redis --> Result2[失败 锁已存在]
    end

锁的安全释放

仅有 SET NX EX 还不够，释放锁时也需要保证安全性。如果一个客户端获取锁后，由于某种原因处理时间过长，锁过期自动释放。此时另一个客户端获取了锁。前面那个客户端完成后释放锁，可能会错误地释放掉后面客户端持有的锁。

NeoTask 使用 Lua 脚本解决这个问题。锁的值设置为客户端的唯一标识，释放锁时先检查当前锁的值是否与客户端标识匹配，只有在匹配时才执行删除操作。整个检查删除过程在 Lua 脚本中原子执行。

锁的数据结构

锁键设计

NeoTask 的锁键采用统一的命名规范。任务锁的键名格式为 neotask🔒任务标识，节点锁的键名格式为 neotask:node:节点标识，批量锁的键名格式为 neotask:batch:批次标识。

锁的值包含持有者信息和时间戳。持有者信息由节点标识和随机字符串组成，确保全局唯一性。时间戳用于记录锁获取时间，便于排查问题。

锁还附带过期时间，默认值为三十秒。看门狗会定期续期，防止长时间任务导致锁过期。

erDiagram
    LOCK {
        string key PK "neotask:lock:{task_id}"
        string value "node_id:random:timestamp"
        int ttl "30秒"
    }

锁的存储位置

所有锁统一存储在 Redis 中。Redis 的单线程模型天然保证了锁操作的原子性，其高性能特性满足分布式锁的低延迟要求。

锁与任务状态存储在同一 Redis 实例中，简化了系统架构，避免跨存储的一致性问题。对于 Redis 集群模式，锁键需要配置哈希标签确保相关键落在同一槽位。

锁的生命周期

锁的获取

锁的获取是原子操作。客户端调用 acquire 方法，传入锁键和期望的 TTL。方法内部执行 SET NX EX 命令。

如果返回成功，客户端获得锁，锁的值设置为客户端标识和当前时间戳。如果返回失败，说明锁已被其他客户端持有，当前客户端需要等待或放弃。

sequenceDiagram
    participant Client as 客户端
    participant Redis as Redis
    
    Client->>Redis: SET lock task:123 node:A NX EX 30
    alt 锁未被占用
        Redis-->>Client: OK
        Client->>Client: 获得锁，开始执行业务
    else 锁已被占用
        Redis-->>Client: nil
        Client->>Client: 锁被占用，等待或放弃
    end

锁的续期

对于执行时间可能超过锁 TTL 的任务，需要自动续期机制。NeoTask 的看门狗负责在任务执行期间定期延长锁的过期时间。

看门狗在锁获取成功后启动，每隔 TTL 的三分之一时间执行一次续期操作。续期操作通过 Lua 脚本检查锁的持有者，确认仍是当前客户端后重置过期时间。

续期失败或客户端主动释放锁时，看门狗停止续期。

sequenceDiagram
    participant WatchDog as 看门狗
    participant Redis as Redis
    participant Lock as 分布式锁
    
    Lock->>WatchDog: 启动看门狗(ttl=30)
    
    loop 每10秒
        WatchDog->>Redis: EXPIRE lock:task 30
        Redis-->>WatchDog: 续期成功
        WatchDog->>WatchDog: 重置计数器
    end
    
    Lock->>WatchDog: 停止看门狗
    WatchDog->>WatchDog: 结束续期循环

锁的释放

任务完成后，客户端主动释放锁。释放操作使用 Lua 脚本，先验证持有者身份，确认是当前客户端后再执行删除。

主动释放是最佳实践，可以减少锁的占用时间，避免等待锁自动过期。即使客户端忘记释放，锁也会在 TTL 后自动过期，保证系统不会死锁。

flowchart TB
    subgraph Release["锁释放流程"]
        Complete[任务完成] --> Check{是否需要释放}
        Check -->|是| Validate[验证持有者身份]
        Check -->|否| Timeout[等待自动过期]
        
        Validate --> Match{身份匹配?}
        Match -->|是| Delete[删除锁键]
        Match -->|否| Skip[跳过释放]
        
        Delete --> End([锁已释放])
        Timeout --> Expire[锁自动过期]
        Expire --> End
        Skip --> End
    end

看门狗机制

为什么需要看门狗

分布式锁的 TTL 是一个两难的选择。TTL 设置过短，任务执行时间稍长就会导致锁提前释放，其他节点可能抢占任务造成重复执行。TTL 设置过长，节点崩溃后锁需要很长时间才能释放，影响系统恢复速度。

看门狗机制解决了这个矛盾。锁的 TTL 可以设置得相对较短（如三十秒），同时看门狗在任务执行期间自动续期。这样既保证了故障时锁能快速释放，又保证了长任务不会因锁过期而中断。

看门狗的实现

看门狗在获得锁后启动，在释放锁后停止。续期间隔通常设置为 TTL 的三分之一，确保障在过期前得到续期。

续期操作本身需要验证持有者身份，避免错误续期。续期失败时，看门狗停止续期，任务执行方应感知到锁已丢失并中断执行。

锁的超时与恢复

锁自动过期

锁的 TTL 确保节点崩溃后锁能够自动释放。TTL 的默认值为三十秒，这是一个平衡故障恢复速度和续期开销的值。

锁过期后，Redis 自动删除该键。其他节点可以重新获取锁，执行原本被占用的任务。这种机制称为任务回收或故障转移。

孤儿任务回收

当节点崩溃且锁过期后，原本由该节点执行的任务成为孤儿任务。NeoTask 的回收器定期扫描运行中状态的任务，检查关联的锁是否存在。

如果锁已过期而任务状态仍为运行中，回收器判断该任务为孤儿任务。根据任务配置的恢复策略，回收器决定是立即重试、延迟重试还是移入死信队列。

stateDiagram-v2
    [*] --> 锁获取: 节点获取锁
    
    锁获取 --> 任务执行: 获取成功
    锁获取 --> 等待重试: 获取失败
    
    任务执行 --> 任务完成: 执行成功
    任务执行 --> 任务失败: 执行失败
    任务执行 --> 节点崩溃: 节点故障
    
    节点崩溃 --> 锁过期: 等待 TTL
    锁过期 --> 任务回收: 回收器扫描
    任务回收 --> 锁获取: 重新入队
    
    任务完成 --> 锁释放: 主动释放
    任务失败 --> 锁释放: 主动释放
    等待重试 --> 锁获取: 延迟后重试

锁的性能优化

批量锁操作

在高吞吐场景下，每个任务都单独获取锁会产生较大的开销。NeoTask 的预取机制结合了批量锁操作。

预取脚本在批量获取任务的同时，为每个任务设置分布式锁。一次网络往返完成了多个任务的获取和锁定，大幅减少了锁操作的开销。

批量锁操作使用同一个 Lua 脚本完成，确保整个批次的原子性。要么整批任务都被当前节点锁定，要么都不锁定。

sequenceDiagram
    participant Prefetcher as 预取器
    participant Redis as Redis
    
    Prefetcher->>Redis: 批量抢占脚本(count=20)
    
    loop 每个任务
        Redis->>Redis: ZRANGE 获取任务
        Redis->>Redis: SETNX 设置锁
        Redis->>Redis: ZREM 移除任务
    end
    
    Redis-->>Prefetcher: 返回成功锁定的任务列表

锁粒度选择

NeoTask 在不同场景使用不同粒度的锁。

任务级锁是最细粒度的锁，每个任务独立锁定。这种粒度并发度最高，但锁操作次数最多。

节点级锁用于节点注册和心跳，粒度较粗，操作频率低。

批量锁用于预取操作，锁住一批任务，减少锁操作次数。

合理选择锁粒度可以在并发和开销之间取得平衡。

    subgraph LockGranularity["锁粒度"]
        Task[任务级锁
细粒度 高并发]
        Batch[批量锁

    end

锁的监控与诊断

锁状态查询

NeoTask 提供了锁状态查询接口，便于排查问题。

is_locked 方法检查锁是否被占用。get_owner 方法获取锁的持有者标识。get_lock_info 方法获取锁的详细信息，包括持有者、获取时间、剩余 TTL 等。

锁扫描与清理

运维场景下，有时需要清理异常残留的锁。NeoTask 提供了锁扫描和批量清理功能。

scan_locks 方法扫描所有匹配模式的锁键，返回键名列表。cleanup_stale_locks 方法清理过期或异常的锁。cleanup_by_owner 方法清理指定持有者的所有锁。

这些方法主要用于运维工具和管理后台，常规业务代码不应调用。

与任务队列的集成

抢占式消费中的锁

在 NeoTask 的去中心化架构中，锁与任务队列紧密集成。节点从队列抢占任务时，通过 Lua 脚本原子地完成出队和加锁操作。

这个设计确保了任务一旦被节点获取，同时就被该节点锁定。其他节点无法抢占同一个任务，避免了重复执行。

flowchart LR
    subgraph Preempt["抢占消费流程"]
        Queue[(任务队列)] --> Script[Lua脚本]
        Script -->|原子操作| Lock[设置锁]
        Lock --> Worker[执行任务]
        Worker --> Complete[释放锁]
    end

延迟任务中的锁

延迟任务在调度阶段不涉及锁，只有在任务从延迟队列转入即时队列并被节点抢占时才获取锁。

延迟任务到达执行时间后，从延迟队列移入优先级队列，等待节点抢占。节点抢占时执行相同的锁获取流程。

周期任务每次执行时都重新获取锁，确保每次执行都是独立的，不会互相干扰。

配置建议

TTL 设置

TTL 的设置需要根据任务执行时间进行调整。

短任务可以使用较短的 TTL，如十五秒。这能加快故障恢复速度，但需要看门狗更频繁地续期。

长任务需要使用较长的 TTL 或依赖看门狗。TTL 六十秒适合大多数场景。

混合场景建议使用三十秒的默认值，配合看门狗自动续期。

重试策略

锁获取失败时，客户端可以选择重试。NeoTask 的重试策略采用指数退避，避免频繁重试造成 Redis 压力。

首次重试延迟一秒，后续重试延迟加倍，最大延迟不超过十秒。重试次数默认三次，超过后放弃获取。

常见问题

锁提前过期

现象是任务执行过程中锁提前释放，其他节点获取锁后重复执行任务。

原因可能有两个。任务执行时间超过锁 TTL，看门狗未正常续期。Redis 连接问题导致续期失败。

解决方案包括适当增加锁 TTL，确保足够覆盖任务执行时间。检查看门狗是否正常启动和运行。确保网络稳定，避免 Redis 连接中断。

锁释放失败

现象是任务完成后锁未释放，后续任务无法获取锁。

原因可能是主动释放时持有者身份验证失败，网络问题导致释放请求丢失。

解决方案是依赖锁的自动过期机制，等待 TTL 后自动释放。排查身份验证逻辑，确保释放时使用正确的持有者标识。添加锁释放重试机制。

锁竞争激烈

现象是大量客户端争抢同一个锁，导致性能下降。

原因可能是锁粒度过粗，多个任务争抢同一个锁。任务并发度过高，超过锁的承受能力。

解决方案是细化锁粒度，按任务或资源类型分别加锁。使用批量锁减少锁操作次数。考虑使用其他协调机制替代锁。

总结

NeoTask 的分布式锁基于 Redis 实现，充分利用了 Redis 的原子操作和单线程模型，提供了高性能、高可用的锁服务。

看门狗机制解决了锁 TTL 与任务执行时间之间的矛盾，既保证了故障快速恢复，又支持长时间任务。批量锁操作减少了锁操作开销，提升了系统吞吐量。

锁扫描与清理功能为运维提供了便利，可以排查和清理异常锁状态。与任务队列的紧密集成保证了抢占式消费的正确性。

通过合理配置 TTL、重试策略和锁粒度，NeoTask 的分布式锁能够满足各种场景的需求，为分布式任务调度提供可靠的并发控制能力。
