如何将一个10分钟短剧转化为分镜脚本

发表于2026-05-14|更新于2026-05-21
|AIAgentPenShot
|浏览量:

10分钟短剧转化为分镜脚本

相关链接：GitHub | 官网 | 文档 （架构设计 · 召回策略 · MCP Server · 一致性策略 · RAG 增强）

准备工作

从GitHub克隆项目后，首先配置环境变量。复制.env.example为.env文件，填入DeepSeek API密钥作为LLM提供商，同时配置Qwen的嵌入模型用于向量化。Redis用于短期记忆缓存，需要确保本地服务已启动。

flowchart LR
    subgraph Setup["环境配置"]
        A[克隆项目] --> B[安装依赖]
        B --> C[配置API密钥]
        C --> D[启动Redis]
        D --> E[验证连接]
    end

准备一份约1200字的短剧剧本，内容为都市情感题材。剧本采用自然语言格式，包含场景描述、角色对话和动作指示。

场景设定

以一个都市情感短剧为例，剧本讲述深夜加班的设计师接到神秘电话后，在雨夜中与失散多年的朋友重逢的故事。全剧时长约3分钟，包含完整的三幕结构：铺垫、冲突、和解。剧本以自然语言撰写，包含场景描述、角色对话、动作指示和情感标注。

原始剧本约1200字，涵盖3个主要场景、2个核心角色，以及完整的情绪弧线从压抑到释放的转变。剧本中包含大量细节描写，如窗外的雨声、键盘敲击声、咖啡杯上的雾气等环境元素，以及角色微表情和肢体语言的变化。

执行过程

通过Python SDK调用PenShot，代码只有几行。创建代理实例后，调用breakdown_script_async方法提交任务，系统返回task_id用于追踪进度。

提交任务后，系统开始执行六阶段工作流。

输入与输出概览

PenShot 接收原始剧本文本后，启动完整的六阶段工作流。输入是未经任何预处理的中文自然语言剧本，输出则是结构化的分镜脚本，包含时间轴规划、镜头序列、AI视频生成提示词、音频提示词以及质量报告。

本次运行中，原始剧本被解析为多个结构化文档块，进而拆分为若干个视觉镜头，最终生成对应数量的视频片段。每个片段都配备了详细的英文提示词和对应的音频描述。总输出数据量可观，包含完整的元数据、片段间连续性约束说明、以及每个片段的风格参数和负面提示词。

各阶段实战记录

剧本解析阶段

剧本解析器首先对原始文本进行格式识别，判断输入类型为自然语言叙事而非标准剧本格式。解析器调用LLM进行语义理解，识别出场景边界、角色登场、对话段落和动作描述。

系统首先识别剧本格式为natural_language，调用DeepSeek模型进行分析。由于历史解析质量评分为0，系统自动启用了增强验证模式，对输出进行更严格的校验。

解析耗时约21秒，最终完整性评分达到0.87。系统从剧本中提取了3个场景、2个角色、3段对话和4个动作。解析结果被向量化为3个文档存入Chroma数据库。

解析过程中，系统启用了增强验证模式，因为历史解析质量评分较低，需要更严格的校验。最终完成度达到较高水平，系统提取了多个场景、主要角色、若干段对话和多个动作序列。解析结果的完整性评分综合考虑了场景连续性、角色一致性和叙事完整性三个维度。

生成的文档随后被向量化并存入知识库，每个文档块都附加了剧本ID作为元数据标签，确保后续查询时的数据隔离。知识库中的嵌入向量维度为特定值，使用BGE嵌入模型生成，支持后续的语义相似度检索。

镜头拆分阶段

镜头拆分器接收结构化剧本后，开始将文本描述转化为视觉语言。系统分析了每个场景中的动作节奏和情感强度，决定镜头的类型和时长。

拆分耗时约8秒，生成7个镜头，总时长19秒。镜头类型分布为特写4个、中景2个、远景1个。随后系统启用了时长增强器，对5个镜头的时长进行了微调，使节奏更加合理。

stateDiagram-v2

    场景分析 --> 镜头类型识别: 提取视觉元素

    时长估算 --> 镜头序列生成: 按时序组织
    镜头序列生成 --> 质量校验: 检查合理性
    质量校验 --> [*]: 输出ShotSequence
    
    state 时长估算 {
        [*] --> 对话场景: 较长时长
        [*] --> 动作场景: 较短时长
        [*] --> 环境场景: 中等时长
    }

拆分结果覆盖了从全景到特写、从室内到室外的多种景别。每个镜头都标注了类型、预估时长和视觉重点。镜头总时长基于剧本中动作密度的分析，动作密集处分配更短的镜头以保持节奏紧凑，情绪沉淀处分配更长的镜头以营造氛围。

镜头类型分布体现了情感驱动型短剧的视觉语言规律：特写镜头用于表现角色情感细节，中景镜头用于对话场景，远景镜头用于环境建立。

视频分割阶段

分割器将镜头进一步切分为AI视频模型可处理的片段。每个镜头的时长被精确分配到合理的范围内，确保符合主流视频生成模型的输入限制。

例如一个起身走向窗边的镜头，被分割为三个片段：从座位站起、转身面向窗户、迈步走向窗边。每个片段时长控制在3到4秒之间。

分割耗时约1秒，生成7个片段，总时长24.33秒，平均每个片段3.48秒。

分割结果生成多个片段，所有片段都在时长限制以内，分布合理。

提示词转换阶段

这是最耗时的阶段，耗时约66秒。系统为每个片段调用LLM生成英文提示词，同时自动生成对应的音频描述。

提示词平均长度达到407字符，最长达到600字符。所有7个片段都配备了音频提示词。内容涵盖了视觉描述、相机运动、光照氛围、负面提示词等完整要素。

转换器为每个片段生成了包含视觉描述、相机运动、光照氛围、画质标签等要素的完整提示词。同时自动生成了对应的音频提示词，包括环境音、角色语气、背景音乐等信息。

提示词长度因片段内容而异，最长的片段提示词包含丰富的细节描述。所有片段都配备了音频提示词，覆盖率达到百分之百。提示词风格统一为电影质感，使用了诸如浅景深、柔光、胶片颗粒等专业术语。

质量审查阶段

质量审查分为规则审查和LLM深度审查两层。规则审查检查多个硬性指标，包括提示词长度是否达标、片段时长是否合规、音频提示词是否存在等。LLM深度审查则从语义层面评估提示词质量。

规则审查检查了5个硬性指标，全部通过获得100分。LLM深度审查发现了4个语义层面的问题，合并各阶段的历史问题后，最终质量评分为40分，评级为major_issues。

系统自动触发了修复流程。剧本解析阶段发现1个问题但没有执行实际修复，提示词阶段发现4个问题也没有执行修复。这说明发现的问题属于警告级别，不影响最终输出质量。

连续性检查阶段

连续性检查器对片段序列进行了全面的连贯性验证，检查维度包括角色服装一致性、场景元素延续性、动作衔接流畅度、光影色调统一性。

连续性检查器对7个片段进行了全面验证，包括角色服装是否一致、场景元素是否延续、动作衔接是否流畅、光影色调是否统一。

检查结果完美通过，发现0个问题。这说明从剧本解析到提示词生成的整个过程中，角色状态、场景信息、视觉风格被完整地传递和保持了。

输出数据结构

最终输出的JSON文件包含完整的片段序列，每个片段都包含以下核心字段：唯一标识符fragment_id、英文视频提示词prompt、中文描述description、负面提示词negative_prompt、预估时长duration、模型适配参数model、视觉风格style、音频提示词对象audio_prompt。

音频提示词对象进一步包含了音频ID、提示词内容、模型类型、音频风格等子字段。这种结构化的输出可以直接对接主流的视频生成模型的API。

遇到的挑战与解决

首次运行时，Redis连接失败导致短期记忆不可用。检查发现.env中Redis配置的数据库编号错误，修正后短期记忆正常初始化。

另一个问题是提示词转换阶段耗时过长。分析发现是因为每个片段都独立调用LLM，没有利用上下文缓存。系统内置的提示词模板缓存机制缓解了这一问题，但根本解决需要等待后续版本的批量处理优化。

质量审查评分只有40分，但连续性问题为0。这说明提示词本身质量有待提升，但角色和场景的一致性保持得很好。后续可以通过调整提示词模板和增加成功案例的学习来改善。

性能表现

整个处理流程耗时在合理范围内。各阶段时间分布呈现出典型的LLM调用特征：提示词转换占据总时间的大部分，这是调用LLM生成高质量提示词的必要开销。

系统在循环控制和重试机制方面表现稳健。每个核心节点都经历了必要的循环次数，表明系统进行了适当的自我修复和优化。全局循环计数远低于上限，留有充足的余量。

记忆系统完整记录了各阶段的中间结果。Redis短期记忆成功初始化并建立了连接，中期记忆保存了各阶段的统计摘要，长期记忆向量库存储了剧本解析结果和模板索引。任务完成后，短期记忆被正确清理，释放了资源。

检查点系统创建了SQLite数据库文件，记录了工作流的执行状态。即使发生意外中断，系统也可以从最近的检查点恢复执行，无需从头开始。
