PenShot 召回策略

深入解析 PenShot 的多维召回体系：从 GraphRAG 到自迭代优化，构建智能剧本分镜的“记忆中枢”

相关链接：GitHub | 官网 | 文档 （架构设计 · 集成指南 · MCP Server · 一致性策略 · RAG 增强）

常用召回方案

方案名称 核心原理 适合场景 技术挑战
GraphRAG 构建知识图谱，通过实体和关系进行检索 人物关系复杂、逻辑因果强的剧本（如悬疑、宫斗）。 图谱构建成本高，数据稀疏时效果不佳。
长记忆模块 隔离存储长期（角色设定）和短期（当前剧情）信息 需要保证跨集数角色性格、说话风格一致性的长剧创作。 需设计有效的记忆总结与压缩机制。
自迭代优化 通过“生成-质检-修复”闭环持续优化 解决生成内容的硬伤（如逻辑矛盾、人设崩塌）导致的“伪召回”。 依赖高质量、多维度的自动化质检标准。
混合检索策略 结合关键词（BM25）、向量、结构化标签的多路召回 需要高精度、高召回率，且兼容模糊语义和精确匹配的通用场景。 多路结果的融合与重排序算法设计复杂。
多智能体协作 不同Agent负责不同维度的检索（如分镜、台词） 分工明确的专业影视制作流水线，各环节对素材需求差异大。 智能体间的通信与任务编排开销大。

方案详解

GraphRAG：让AI真正“读懂”剧本逻辑

传统RAG基于向量相似度，容易忽略剧情中复杂的逻辑关系。阿里云AnalyticDB的方案引入了GraphRAG，它在向量检索的基础上，构建了包含“角色-事件-场景”的知识图谱)。

  • 用法：当用户要求“写一段男主发现助手背叛后的对峙戏”时，系统不仅找“背叛”相关语义，还能图谱中定位“男主”和“助手”的关系及历史。
  • 价值：确保后续情节发展与前面埋下的伏笔和人物关系逻辑自洽。

长记忆模块：确保人设“不崩塌”

剧本创作的最大痛点是角色或场景的“不一致”。AnalyticDB for PostgreSQL内置的Long-term Memory技术就是为此设计。

  • 分层存储：将长期记忆（如角色性格、外貌、核心世界观）与短期记忆（如当前场景对话、情绪）分开管理)。
  • 效果：在生成新对话时，AI会强制参考长期记忆库中的角色设定，有效防止角色“精分”。

框架仿写与结构化提取：迁移“神”而非“形”

爱奇艺“纳逗Pro”智能体及阿里云方案均强调了结构化提取的重要性)。

  • 做法：从参考剧本中提取出叙事框架（如起承转合的结构、冲突模式、情感线），将其剥离具体台词后存入向量库)。
  • 应用：当用户想写一个“现代版罗密欧与朱丽叶”时，系统能召回的是两个家族世仇下的爱情冲突框架，而非直接给古代台词，实现了高层次的创意复用。

自迭代优化：用“质检”倒逼“召回”

有时召回失败不是因为没找到，而是因为生成的内容质量不过关（如角色做出了不符合人设的动作）。这就引入了质检闭环：

  • 流程：生成 → 自动质检（检查角色、逻辑、格式等10+维度） → 发现缺陷（如“逻辑一致性”低分） → 触发定向召回（专门去找前文中相关的逻辑设定） → 修复。
  • 意义：通过反馈机制，让系统知道自己“漏掉了什么”，实现自我进化。

本智能体短期支持短剧单次交互，长期支持电影/动漫/电视剧长剧本”的需求，需要选择一套具备平滑演进能力的技术方案。核心原则是：短期方案作为长期的组件或简化版，避免未来重构。

单次交互模式（短剧）

短剧（单集2-5分钟，总时长20-60分钟）的核心需求是快速响应和成本可控。

组件 技术选择 说明
查询理解 关键词(BM25) + 轻量向量检索 双路召回，平衡精度与召回率
记忆管理 单次会话上下文(同场景内) 无需长期记忆，降低复杂度
智能体架构 单智能体或双智能体(剧本+分镜) 简化调度，端到端效率优先
召回优化 实时反馈调整 生成不满意→触发重新召回
成本控制 模型轻量化(LightX2V蒸馏技术) 5秒视频生成<5秒，单集成本500-1000元

超长剧本支持（电影）

电影/电视剧/动漫（单集40分钟以上，总时长数小时）的核心挑战是长上下文管理和跨集一致性。

万字级剧本 → 语义索引系统 → 事件图谱构建 → 分层检索

  • 采用语义索引管道：将长篇内容分割为5-15分钟的重叠窗口，提取层次化摘要

  • 章节事件图谱：用RAG技术分析长文本结构，提取事件、动作和情感点

  • 参考ViMax的智能长脚本引擎：支持小说级内容自动分段

组件 方案 作用
GraphRAG 知识图谱 + 向量检索 理解人物关系、因果逻辑
长记忆存储 结构化记忆库 角色设定、场景资产、世界观
多级缓存 热/温/冷三层 高频角色设备→快速检索
混合检索 关键词+向量+结构化标签 多路召回+重排序
质检闭环 生成→检查→修复→优化 自我进化的反馈机制

关键数据流

  1. 主创作流（蓝色实线）：用户输入 → 导演拆解 → 各Agent创作 → 质检 → 输出
  2. 记忆流（绿色虚线）：创作过程中持续读写记忆库，保持一致性
  3. 反馈流（红色虚线）：质检发现缺陷 → 触发导演重新调度或定向召回
  4. 仲裁流（橙色点线）：导演Agent随时干预风格冲突或资源分配问题

设计目标

PenShot 召回系统针对剧本分镜场景的特殊需求，设定了以下核心设计目标：

  • 高召回率
    在剧本要素（角色、场景、道具）的识别上追求95%以上的召回率，确保不会遗漏关键信息。剧本中任何一个角色、场景或重要道具的遗漏都可能导致后续分镜生成的严重偏差。

  • 低延迟响应
    整体召回流程控制在200ms以内，其中精确匹配<10ms，向量检索<100ms，图谱推理<200ms。这保证了用户交互的流畅性，避免长时间的等待影响创作体验。

  • 语义理解准确
    支持中文剧本的深度语义理解，能够区分“角色在说话”和“角色被提及”等不同语境。例如，“陈阳说…”和“陈阳被提到…”需要有不同的召回策略和处理方式。

  • 逻辑关系感知
    能够理解角色间的互动关系、事件的因果链条、场景的转换逻辑。这对于长剧本的连贯性至关重要，确保生成的视频片段在叙事逻辑上自洽。

  • 跨剧本知识迁移
    从历史成功的剧本中学习模式，应用到新剧本的创作中。通过记录和分析高质量剧本的结构特征、节奏规律，系统能够为新剧本提供更有价值的参考建议。

权重说明
高权重策略（精确匹配、语义向量、记忆系统）在融合排序中占主导地位，因为它们直接关系到分镜质量的核心要素。中权重策略（关键词、知识图谱）作为补充，提供额外的召回路以应对边缘情况。

混合检索策略详解

核心解决的问题

剧本分镜场景中，用户查询呈现多样化的特点：有时需要精确的角色名称匹配（如“林小雨的所有台词”），有时需要语义相似的内容（如“雨天的氛围场景”），有时需要结构化的筛选（如“第三幕的对话”）。单一的检索方式无法同时满足所有这些需求。

混合检索通过多路并行召回+融合排序的方式，综合各种检索策略的优势：

需求类型 示例查询 适用策略 原因
精确匹配 “查找陈阳的所有台词” 关键词BM25 实体名称需精确命中
语义相似 “类似雨中相遇的浪漫场景” 向量检索 匹配意境而非字面
关系推理 “男主发现真相后的反应镜头” GraphRAG 需理解因果链
结构筛选 “第二幕中时长>3秒的特写” 标签过滤 结构化条件查询

融合算法说明

倒数排序融合（RRF）
RRF是混合检索中最常用的融合算法。它的核心思想是：对每个召回源的结果独立排序，然后计算每个文档的RRF分数。

公式为：RRF_score = Σ 1/(k + rank_i)

其中k是常数（通常取60），rank_i是文档在第i个召回源中的排名。RRF的优势在于不需要对分数进行归一化，不同召回源的分数尺度差异不影响融合效果。

加权求和融合
当各召回源的分数已经归一化到同一尺度时，可以采用加权求和：final_score = Σ w_i * score_i

权重的设定基于经验调优和各策略的历史表现。在PenShot中，精确匹配和向量检索获得较高权重，因为它们对分镜质量影响最大。

学习排序（LTR）
通过收集用户反馈和人工标注数据，训练一个排序模型来学习最优的融合权重。这是系统自迭代能力的重要组成部分，随着使用数据的积累，融合策略会持续优化。

GraphRAG知识图谱

核心解决的问题

传统RAG（检索增强生成）基于向量相似度进行检索，这种方式难以理解剧情中的复杂逻辑关系。例如：

场景一：剧本中“男主”和“助手”在前三集建立了深厚的信任关系，第五集“助手背叛”。当用户要求生成“男主发现助手背叛后的对峙戏”时，传统RAG只能检索到“背叛”相关的片段，但无法理解“为什么背叛会带来强烈的情绪冲击”——这需要理解前文中建立的信任关系。

场景二：用户要求“延续上一场的氛围”，传统RAG无法理解“上一场”指的是哪个场景，更无法理解该场景的氛围特征是什么。

GraphRAG通过在向量检索的基础上构建知识图谱来解决这些问题。图谱中包含“角色-事件-场景”等实体，以及它们之间的各种关系（互动、因果、时序等）。

三层记忆的职责划分

记忆层 存储内容 容量限制 生命周期 淘汰策略
短期记忆 当前会话的对话历史、临时状态 20条 1小时 LRU + TTL
中期记忆 阶段总结、关键决策记录 500 tokens 整个任务周期 摘要压缩
长期记忆 角色设定、场景模板、学习模式 无限制 永久 向量索引

一、短期记忆详解

短期记忆存储当前会话中的交互历史，包括：

  • 用户最近3-5轮的查询和反馈

  • 当前正在处理的片段状态

  • 临时的变量和标志位

短期记忆采用LRU（最近最少使用）淘汰策略，当容量达到上限时，移除最久未访问的条目。同时设有TTL（生存时间）机制，超过1小时的条目自动过期。

二、中期记忆详解

中期记忆在每个处理阶段结束时生成摘要。例如，剧本解析完成后，系统会生成一个摘要：

  • 识别出的角色列表及其特征

  • 场景的数量和类型分布

  • 发现的问题和已执行的修复

这些摘要被压缩存储（最大500 tokens），在整个任务周期内可用。当任务完成时，中期记忆可以选择性地归档到长期记忆或清理。

三、长期记忆详解

长期记忆存储跨任务的持久化知识，包括：

  • 角色设定库：从历史剧本中提取的角色特征模式

  • 场景模板库：高质量的场景描述模板

  • 成功案例库：经过质量审计的高分片段

  • 学习模式：从修复历史中总结的规律

长期记忆通过向量索引实现语义检索，目前已存储78条文档，使用768维的BGE嵌入模型。

自迭代优化闭环

核心解决的问题

传统的召回系统是静态的——检索策略一旦确定就不会改变。但剧本分镜场景中，用户的创作风格和质量标准会随着使用而演进。自迭代优化通过“生成-质检-修复”闭环来解决这个问题。

具体场景：

  • 系统生成了一个分镜，质量审计发现“角色情绪描述不一致”
  • 系统分析问题根源：召回时没有考虑角色在之前的情绪状态
  • 系统调整检索策略，增加“角色情绪历史”作为检索维度
  • 下次遇到类似查询时，系统会额外召回该角色的情绪变化轨迹

这个闭环让系统能够从自己的错误中学习，实现自我进化。

质检维度详解

PenShot的自动质检覆盖10+个维度，每个维度都有明确的检查规则和严重程度分级：

  • 角色一致性
    检查生成内容中的角色行为、对话、情绪是否与既定的角色设定一致。例如：内向角色是否被描述为“外向开朗”，学生角色是否被赋予“教授”的台词风格。

    检查方式：将生成内容中的角色描述与长期记忆中存储的角色设定进行语义比对，计算偏离度。

  • 场景连续性
    检查场景转换是否自然，是否有逻辑跳转。例如：角色在室内突然出现在室外却没有过渡镜头。

    检查方式：分析相邻片段的场景ID和描述，检测是否有不合理的场景切换。

  • 动作逻辑性
    检查动作序列是否符合物理规律和叙事逻辑。例如：角色“拿起书”之后才能“翻页”，“站起来”之后才能“走向门口”。

    检查方式：构建动作依赖图，验证生成的动作序列是否满足前置条件。

  • 时间连续性
    检查时间线是否连贯，时间跨度是否合理。例如：前一个片段是“清晨”，后一个片段变成“深夜”却没有时间过渡说明。

    检查方式：提取每个片段的时间标记，检测时间跳变是否超过合理阈值。

  • 视觉风格一致性
    检查所有片段的视觉风格描述是否统一。例如：一个片段要求“暖色调”，另一个要求“冷色调”，且没有风格转换的说明。

    检查方式：提取风格关键词（色调、光线、色彩等），计算片段间的风格差异。

  • 道具一致性
    检查道具的状态变化是否符合逻辑。例如：“书”被雨水浸湿后应该是“湿的”，后续片段不能描述为“干燥的”。

    检查方式：建立道具状态机，追踪每个道具的状态变化。

自迭代优化流程详解

步骤一：问题发现

质量审计系统对生成结果进行多维度的自动检查，输出：

  • 问题类型（角色不一致、场景跳转等）

  • 严重程度（错误、警告、提示）

  • 问题位置（哪个片段、哪个属性）

  • 问题描述（具体哪里出了问题）

步骤二：根源分析

系统分析问题的根本原因，可能的原因包括：

  • 召回阶段：检索到的相关信息不足以支撑正确生成

  • 生成阶段：LLM推理错误或指令理解偏差

  • 设定缺失：角色或场景的设定信息不完整

步骤三：定向召回

根据问题类型触发定向召回策略：

  • 角色一致性问题：额外召回该角色的完整设定和过往表现

  • 场景连续性问题：额外召回前后场景的描述和氛围设定

  • 动作逻辑问题：额外召回动作依赖关系和状态变化

  • 道具一致性问题：额外召回道具的状态变更历史

步骤四：重新生成

使用增强后的召回结果重新生成内容，再次进行质量审计，直到通过或达到重试上限。

步骤五：经验学习

将成功的修复模式记录到长期记忆中：

  • 问题特征（哪些条件触发了问题）

  • 修复策略（采取了什么检索调整）

  • 修复效果（质量分数的提升）

  • 适用场景（哪些类似情况可以复用）

    Parser->>Memory: 存储解析结果
    
    Parser->>Segmenter: 传递角色+场景信息
    
    Segmenter->>Segmenter: 生成分镜序列
    Segmenter->>Memory: 存储分镜结果
    
    Segmenter->>Splitter: 传递分镜序列
    
    Splitter->>Splitter: 分割长镜头
    Splitter->>Memory: 存储分割结果
    
    Splitter->>Converter: 传递片段信息
    
    Converter->>Converter: 生成提示词
    Converter->>Memory: 检索相似模板
    
    Memory-->>Converter: 返回历史模板
    
    Converter->>Converter: 融合生成最终提示词
    Converter->>User: 输出结果
