分布式任务调度（NeoTask）延时任务

NeoTask 延时任务

📚 返回首页 | 📖 快速上手 | 📘 API 参考 | ⚙️ 配置说明 | 🐙 GitHub | 🐱 NeoTask 官网

概述

在分布式系统中，延时任务（Delayed Tasks）是一种常见的需求，例如：订单30分钟未支付自动取消、用户注册7天后发送回访邮件、定时发送报表等。实现可靠的延时任务调度，是任务调度系统从“即时处理”迈向“全场景覆盖”的关键一步。

NeoTask 从 v0.3 版本开始，正式引入了完整的定时调度能力，包括：延时执行、指定时间点执行、固定间隔周期执行以及 Cron 表达式支持。本文将重点介绍延时任务的架构设计与实现原理。

延时任务的核心挑战

与即时任务不同，延时任务面临几个关键挑战：

  • 时间精度：需要保证在指定延迟后准确触发，误差应在可接受范围内

  • 持久化：系统重启后，延时任务不能丢失

  • 高并发下的性能：可能有大量延时任务同时等待（如秒杀活动的倒计时）

  • 分布式环境下的协调：多节点部署时，同一个延时任务不能被重复触发

为了解决这些问题，NeoTask 设计了一套以 时间轮（Time Wheel） 为核心，结合 Redis Sorted Set 持久化的延时任务调度架构。

整体架构设计

延时任务是 NeoTask 定时调度能力的一部分，与即时任务和周期任务共同构成了完整的功能体系。

时间轮（Time Wheel）

最直观的延时任务实现方式是使用 asyncio.sleep(delay) 或为每个任务创建一个定时器。但这种方式存在明显的性能问题：

  • 每个延时任务都需要一个独立的定时器，内存开销大

  • 大量定时器同时存在时，调度开销呈线性增长

  • 系统重启后，所有内存中的定时器都会丢失

时间轮算法通过“空间换时间”的策略，将所有延时任务组织在一个环形数组 + 时间槽的数据结构中，实现了 O(1) 的任务添加和移除复杂度。

    T10 -->|放入| S10
    T65 -->|round=1| S5
    P -->|每秒移动一格| S0

如上图所示，时间轮的核心要素包括：

  • 槽位（Slot）：时间轮被划分为 N 个槽位（NeoTask 默认 60 个），每个槽位代表一个时间单位（1 秒）

  • 指针（Pointer）：每个时间单位移动一格，指向当前需要处理的槽位

  • 轮次（Round）：当任务的延迟时间超过一圈时，记录剩余圈数，每经过一圈减 1

  • 任务链：每个槽位可以挂载多个任务（通过链表或数组）

当一个延时任务被提交时，系统根据其延迟时间计算出目标槽位和轮次，然后放入对应的槽位。时间轮指针每秒移动一格，处理当前槽位中轮次为 0 的任务，并减少其他任务的轮次数。

时间轮的优点

特性 说明
O(1) 插入 无论有多少延时任务，添加新任务的时间复杂度恒定
O(1) 删除 取消延时任务同样高效
批量处理 同一时间槽的任务可以批量执行
内存高效 无需为每个任务创建独立定时器

时间轮实现

在 v0.3 版本中，NeoTask 实现了基于内存的时间轮，用于高性能延时任务调度：

分布式持久化：Redis

纯内存的时间轮虽然性能极高，但无法应对节点重启或故障场景。NeoTask 结合 Redis Sorted Set 实现了延时任务的持久化存储，保证任务不丢失。

  • 延时队列键：queue:delayed（Redis Sorted Set）

    • Member：task_id
    • Score：执行时间戳（Unix timestamp）

  • 任务详情键：task:{task_id}（Redis Hash）

    • 存储任务的完整信息：数据、优先级、状态、创建时间等

扫描调度机制

flowchart TB
    START[扫描循环开始] --> GET_NOW[获取当前时间戳]
    GET_NOW --> QUERY[ZRANGEBYSCORE 查询到期任务]
    QUERY --> HAS_TASK{有到期任务？}
    
    HAS_TASK -->|是| FOR_EACH[遍历任务列表]
    FOR_EACH --> CHECK_LOCK{尝试获取分布式锁}
    CHECK_LOCK -->|成功| MOVE[移入即时队列]
    CHECK_LOCK -->|失败| SKIP[跳过（其他节点已处理）]
    MOVE --> UPDATE[更新任务状态]
    UPDATE --> NEXT[继续下一个]
    SKIP --> NEXT
    NEXT --> HAS_TASK
    
    HAS_TASK -->|否| WAIT[等待下一扫描周期]
    WAIT --> START

分布式环境下的协调

在多节点部署时，每个节点都有独立的时间轮扫描器。它们都会从 Redis 延时队列中获取到期任务。为了避免同一个任务被多个节点重复执行，NeoTask 采用了以下机制：

  1. 原子操作：使用 Lua 脚本或 Redis 事务，确保“查询-移除-加锁”操作原子执行

  2. 分布式锁：在任务被移入即时队列前，先尝试获取该任务的分布式锁（SET NX EX）

  3. 幂等性保证：任务执行前检查状态，避免重复执行

sequenceDiagram
    participant NodeA as 节点A
    participant NodeB as 节点B
    participant Redis as Redis

    Note over NodeA,Redis: 同一时刻，多个节点扫描
    NodeA->>Redis: ZRANGEBYSCORE 获取到期任务
    NodeB->>Redis: ZRANGEBYSCORE 获取到期任务
    
    Redis-->>NodeA: [task-001, task-002]
    Redis-->>NodeB: [task-001, task-002]
    
    NodeA->>Redis: SET lock:task-001 NX EX 30
    NodeB->>Redis: SET lock:task-001 NX EX 30
    
    Redis-->>NodeA: OK（获取锁成功）
    Redis-->>NodeB: NULL（锁已被占用）
    
    NodeA->>Redis: ZREM queue:delayed task-001
    NodeA->>Redis: ZADD queue:priority task-001
    
    Note over NodeB: 跳过已被锁定的任务

时间轮 vs 定期扫描

NeoTask 在延时任务实现上采用了“两级调度”架构：

层级 组件 职责 特点
一级调度 内存时间轮 精确计时，O(1) 复杂度 高性能，但易失
二级调度 Redis Sorted Set 扫描 持久化存储，故障恢复 可靠，但有扫描延迟

性能考量与实践建议

时间槽数量选择

场景 推荐槽位数 说明
高精度需求（秒级） 60-360 每秒一个槽位
分钟级精度 60 每分钟一个槽位
大量长延时任务 减少槽位，增加轮次 减少内存占用

扫描间隔配置

生产环境最佳实践

  1. 合理设置扫描间隔：精度要求不高时，适当增大间隔可减少 Redis 负载

  2. 启用时间轮：高并发场景下，内存时间轮可大幅提升性能

  3. 任务幂等性：确保业务逻辑支持重复执行，应对可能的故障恢复

  4. 监控延时队列积压：设置告警，及时处理调度延迟

总结

延时任务是分布式任务调度系统的核心能力之一。NeoTask 通过 内存时间轮 + Redis Sorted Set 持久化 的两级调度架构，在保证高性能的同时，实现了任务的持久化和故障恢复能力。

  • v0.3 版本核心贡献：奠定了定时调度的基础架构

  • 后续演进：v0.4 引入分布式锁解决多节点并发问题，v1.0 加入看门狗机制防止长延时任务锁过期

  • 未来展望：在 v1.5 版本的任务编排中，延时任务将成为 DAG 工作流的重要组成
